## Phase 9: RAG suggested feedback for lab assistants

- [ ] Complete

**Goal:** When a lab assistant is assigned to a student, surface 2–3 bullet-point coaching suggestions generated by an LLM, grounded in that student's actual submission history and AI feedback via a ChromaDB RAG pipeline. Architecture follows the hybrid design in [[RAG Architecture]].

### Architecture

Two-layer retrieval following Dr. Batmaz's design (must be acknowledged in dissertation):

1. **Pre-filter by student_id** — restrict retrieval to the assigned student's rows only (`df[df.user == student_id]`)
2. **ChromaDB semantic search** — embed Q&A + AI feedback strings; query for most conceptually relevant chunks, filtered by `student_id` metadata

LLM prompt: student struggle metrics + retrieved Q&A snippets → 2–3 actionable bullet points for the assistant. See [[RAG Architecture]] for full pipeline diagram and credit note.

### New dependencies (add to `requirements.txt`)

- `chromadb`
- `sentence-transformers`

### New/modified files

**`code/learning_dashboard/config.py`**

```python
RAG_EMBEDDING_MODEL: str = "all-MiniLM-L6-v2"   # sentence-transformers, open-source, no API cost
RAG_CHROMA_PATH: str                              # path under data/ for persistent ChromaDB storage
RAG_SUGGESTION_MAX_RESULTS: int = 5              # top-k retrieved Q&A chunks per query
RAG_SUGGESTION_CACHE: dict = {}                  # in-process cache keyed by student_id; cleared on session change
```

**`code/learning_dashboard/analytics.py`**

- `build_rag_collection(df, session_id) -> chromadb.Collection`
  - `chromadb.PersistentClient(path=RAG_CHROMA_PATH)`, collection name `session_{session_id}`
  - Document per row: `"{question} | {student_answer} | {ai_feedback}"`
  - Metadata per row: `student_id`, `question`, `incorrectness` (float)
  - Embeddings via `SentenceTransformer(RAG_EMBEDDING_MODEL).encode(docs)`
  - Rebuild guard: only recreate collection when df row count changes (check against stored count in session_state)
- `generate_assistant_suggestions(student_id, df, struggle_row) -> list[str]`
  - Check `RAG_SUGGESTION_CACHE`; return cached result if present
  - Layer 1: filter df to `df[df.user == student_id]`; if < 2 rows, return `[]`
  - Layer 2: ChromaDB query with `where={"student_id": student_id}`, retrieve top-k chunks ordered by incorrectness metadata
  - Build OpenAI prompt:
    - System: `"You are advising a lab assistant on how to help a student. Be concise and practical."`
    - User context: struggle score, struggle level, top-3 question incorrectness %, retrieved Q&A snippets
    - Instruction: `"Give 2–3 short bullet points (max 15 words each) on what to check or discuss with this student."`
  - Call `openai.chat.completions.create(model=config.OPENAI_MODEL, temperature=0, ...)`
  - Parse response into `list[str]` (one bullet per item); store in `RAG_SUGGESTION_CACHE[student_id]`
  - Return `[]` on any error — never break the UI

**`code/learning_dashboard/assistant_app.py`**

- Call `build_rag_collection(df, session_id)` once per session (guard with `st.session_state["_rag_built"]` flag)
- Call `generate_assistant_suggestions(...)` the first time an assistant is assigned a student
- Store suggestions in `st.session_state["cached_suggestions"]` — do not re-call on 5-second auto-refresh
- Render in the assigned-student card, beneath the struggle badge:
  - `st.markdown("**Suggested Focus Areas**")` + `st.markdown(f"• {s}")` for each bullet
  - `st.spinner("Generating suggestions...")` during first load
  - Show `"Not enough data yet"` if student has < 2 submissions
  - Show nothing (no error message) if suggestion generation fails

### Edge cases and error handling

- `chromadb` or `sentence-transformers` not installed → log warning, skip suggestion block silently
- Student has < 2 submissions → skip LLM call, show `"Not enough data yet"` in UI
- OpenAI call fails or returns malformed output → return `[]`, no visible error shown
- Session ends or session_id changes → clear `RAG_SUGGESTION_CACHE` to prevent stale suggestions

### Phase 9 verification

- Join a session with active student data, get assigned to a struggling student → "Suggested Focus Areas" appears with 2–3 bullets
- Wait for the 5-second auto-refresh → suggestions do not reload (verify via log/print)
- Assign a student who has only 1 submission → "Not enough data yet" message shown
- Syntax check: `python -m py_compile code/lab_app.py code/learning_dashboard/analytics.py code/learning_dashboard/config.py`

### Dissertation notes

- ChromaDB collection persists across process restarts within the same session (uses `PersistentClient`)
- Embedding model runs locally — no additional API cost beyond the existing OpenAI LLM call
- Dr. Batmaz designed the hybrid two-layer RAG architecture — must be acknowledged in dissertation (see [[RAG Architecture]])
- Add RAG + ChromaDB to the literature review and Ch4 advanced features section

## Phase 11: In-app Help system (assistant app)

- [ ] Complete

**Goal:** Add a lightweight, mobile-friendly Help panel to the lab assistant app so that a new assistant can understand the interface without needing external documentation. The assistant app runs on phones and the UI must stay compact — Help content should be brief, plain-English, and accessible from every app state without disrupting the live workflow. This is distinct from the instructor dashboard Help (Phase 10); do not carry over the layered maths or Model Guide content here.

### Phase 11 modified files

- `code/learning_dashboard/assistant_app.py` — add Help entry point and render logic
- `code/learning_dashboard/ui/theme.py` — any mobile CSS additions for Help panel styling; must match existing `get_mobile_css()` style

### Design constraints

- The app auto-refreshes every 5 seconds; the Help panel must not trigger an unwanted state reset on refresh (use `st.session_state` to track open/closed state)
- Panel must be accessible from all four app states: join screen, unassigned view, assigned view, session-ended view
- Content must be readable on a small phone screen — short paragraphs, bullet points, no wide tables
- Do not replicate the instructor dashboard's Model Guide or maths documentation here; keep it operational
