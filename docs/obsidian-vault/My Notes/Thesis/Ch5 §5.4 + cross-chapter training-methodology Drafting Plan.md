## Insertion 5 — Ch2 §2.2.4 "Empirical Refinement of Composite Models" (~170 words)

**Anchor**: under the §2.2.4 stub header inserted between [`requirements-specification.tex:137 and 139`](../../../../Report/main-sections/requirements-specification.tex#L137).

```latex
Where the heuristic weight schemes reviewed above dominate the operational learning-analytics literature, a parallel methodological tradition replaces hand-set weights with supervised-fit weights, typically logistic regression or gradient boosting trained against externally-collected labels of student state. The supervisory signal need not be human-annotated: recent work on large language models as second-opinion raters demonstrates substantial agreement with human annotators on classification tasks at a small fraction of the cost \cite{gilardiChatGPTOutperformsCrowd2023, chiangCanLargeLanguage2023, zhengJudgingLLMasaJudgeMTBench2023}, with inter-rater agreement quantified through Cohen's $\kappa$ \cite{CoefficientAgreementNominal} and interpreted against the bands set out by Landis and Koch \cite{landisMeasurementObserverAgreement1977}. Where the weight space is small but contains scalar hyperparameters tuned by trial and error, modern Bayesian optimisation libraries such as Optuna's Tree-structured Parzen Estimator \cite{bergstraAlgorithmsHyperParameterOptimization2011, akibaOptunaNextgenerationHyperparameter2019} produce stable optima in fewer trials than grid or random search. The empirical-refinement work reported in Chapter~\ref{sec:eval-results} adopts all three of these techniques to re-fit the v1 composite weights.
```

**Prose**:

```latex
The deployed v1 composite weights of Sections~\ref{sec: student struggle} and~\ref{sec: task difficulty} were tuned by trial and error in the absence of labelled ground-truth data, as noted above. To close that loop empirically without re-instrumenting the deployed system, the design adopts a self-contained offline pipeline that re-fits the composite weights against second-opinion labels obtained from a large language model rater (GPT-4o-mini), evaluates the refined v2 vectors against the same retrospective protocol the deployed v1 system is evaluated under, and exposes the result as a runtime toggle so the v1 vectors remain the deployed default.

The pipeline is structured in five phases with frozen data hand-offs between them, as Figure~\ref{fig:v2-pipeline-design} shows. Phase~1 hits the submission API once and freezes the response to a Parquet cache so downstream phases are network-free and deterministic; Phase~2 draws a stratified sample of 1{,}306 snapshots across session, cutoff, and band so the training data are not dominated by the largest band; Phase~3 invokes the LLM rater in three modes (struggle, difficulty, and a blind author self-labelling on a fifty-snapshot subset that supports a Cohen's $\kappa$ calibration); Phase~4 fits the three composite weight vectors under L2-regularised logistic regression and the two scalar hyperparameters under Bayesian optimisation; Phase~5 produces the figures and tables of Chapter~\ref{sec:eval-results} from the four resulting JSON artefacts.

Two design decisions are worth flagging in advance of Chapter~\ref{sec:implementation}. First, every phase outputs a frozen on-disk artefact (Parquet for raw data, JSON for snapshots, labels, weights, and hyperparameters), so the pipeline is restart-safe at any phase boundary and any downstream phase can be re-run without recomputing what precedes it. Second, the trained v2 vectors are deployment-ready: the runtime toggle in the V2 stack reads the same JSON files that Phase~4 writes, with no schema translation step between training output and runtime input.
```

## Insertion 6 — Ch4 §4.7.5 "V2 Empirical Refinement Pipeline" (~520 words; opening updated 2026-05-25 with cross-chapter back-refs)

**Anchor**: under the §4.7.5 stub header inserted before [`implementation.tex:1007`](../../../../Report/main-sections/implementation.tex#L1007). This subsubsection covers the **operational implementation** of the pipeline (scripts, code, file artefacts). The **pipeline design** (data flow, key design decisions) lives in Section~\ref{sec:v2-pipeline-design} (Ch3 §3.5, Insertion 8); the **methodology rationale at the evaluation level** lives in Section~\ref{sec:eval-v2-methodology} (Ch5 §5.1.5, Insertion 7).

```latex
The v1 composite weights described in Sections~\ref{sec:analytics} and~\ref{sec:advanced} were tuned by trial and error in the absence of labelled ground-truth data, as Chapter~\ref{sec:design} explains. The v2 stack adds a self-contained offline pipeline whose design is set out in Section~\ref{sec:v2-pipeline-design} and whose evaluation-side methodology is summarised in Section~\ref{sec:eval-v2-methodology}; this section describes the operational implementation. The pipeline re-fits the v1 weights against second-opinion labels obtained from a large language model rater (GPT-4o-mini); the resulting weight vectors live as JSON artefacts under \texttt{data/eval/} and are loaded at request time by the toggle mechanism described earlier in this section. The pipeline runs entirely outside the live request path, so the dashboard's per-request cost is unchanged; v2 vectors are simply read from disk on the first request after a settings change.

The pipeline is structured as five sequential phases, each implemented as a Python script under \texttt{scripts/} or as a Jupyter notebook under \texttt{notebooks/}. Phase~1 (\texttt{eval\_fetch.py}) hits the Loughborough submission API once and caches the full 42{,}443-row response to a Parquet file under \texttt{data/eval/submissions.parquet}; downstream phases never re-hit the API. Phase~2 (\texttt{eval\_common.py}) stratifies the cached submissions into 1{,}306 evaluation snapshots across twenty-one healthy sessions, twelve cutoff times per session, and the four struggle bands, plus seventy-two unique questions for the difficulty side. Phase~3 (\texttt{eval\_label.py}) calls GPT-4o-mini for each snapshot to obtain a binary intervene flag and a four-band ordinal severity for the struggle side, and a four-band difficulty rating for the question side; 1{,}378 labels are produced across 277 batched API calls at an approximate cost of \$0.20.

Phase~4 (\texttt{optimise\_v2\_weights.py} and \texttt{optimise\_hyperparams.py}) fits the v2 vectors. The seven-signal struggle composite and the three-weight improved-blend are fitted under L2-regularised logistic regression --- chosen to preserve the linear functional form of the v1 composite so that v1 and v2 vectors are directly comparable term by term, with a gradient-boosting fallback available had the LR pooled AUC fallen below a held-out threshold of $0.55$, which was not triggered on any of the three composites --- with the regularisation strength \texttt{C} tuned by inner cross-validation on each outer fold; the outer scheme is \texttt{GroupKFold(n\_splits=5)} with session as the group identifier, so train and test partitions never share students. The five-signal difficulty composite is fitted under leave-one-out cross-validation on the seventy-two questions, the only honest scheme at that sample size. The two scalar hyperparameters carried over from Chapter~\ref{sec:design} (the Bayesian shrinkage constant $K$ and the collaborative-filtering similarity threshold $\tau$) are tuned in two independent Optuna~\cite{akibaOptunaNextgenerationHyperparameter2019} studies of fifty trials each using the Tree-structured Parzen Estimator~\cite{bergstraAlgorithmsHyperParameterOptimization2011}, with session-grouped five-fold cross-validation AUC against the intervene labels as the maximisation objective.

Phase~5 (\texttt{notebooks/eval\_main.ipynb}) reads all four JSON artefacts and produces both the figures shown in Chapter~\ref{sec:eval-results} and the auto-generated tables in \texttt{data/eval/results.md}, which serves as the canonical source for the numbers reported in that chapter. The random seed throughout is forty-two, so the snapshot stratification, the cross-validation splits, and the Optuna search are all reproducible from a single command sequence.
```

## Insertion 7 — Ch5 §5.1.5 "V2 Training Methodology" (~210 words; trimmed 2026-05-25 per Option A scope split)

```latex

## §5.4.2 Inter-rater Agreement (~150 words)

```latex

Fifty snapshots drawn at random from the full 1{,}306 were rated by the author through a blind command-line interface in the order the LLM rater had previously seen them, each scored on the four-band scale (On Track / Minor Issues / Struggling / Needs Help) together with a binary intervene flag. Intervene rates are 60\% (human) and 66\% (LLM); the LLM is slightly more inclined to escalate. ## §5.4.10 Hyperparameter Optimisation (~210 words)

```latex
Two scalar hyperparameters were tuned independently of the weight vectors using Optuna's Tree-structured Parzen Estimator \cite{bergstraAlgorithmsHyperParameterOptimization2011, akibaOptunaNextgenerationHyperparameter2019}, maximising session-grouped five-fold cross-validation AUC against the same LLM intervene labels; each study ran 50 trials over its own search range. TPE was preferred over grid or random search as the two search spaces (integer $K$, real $\tau$) are small enough for Bayesian methods to converge in tens of trials where a uniform sweep would require an order of magnitude more. The Bayesian shrinkage constant $K$ moves from v1 default $5$ to v2 best $1$, with CV AUC changing from $0.799$ to $0.805$ --- $\Delta \approx +0.007$, within the per-fold standard deviation, so the result is a robustness validation of the hand-set value rather than an empirical refinement. The collaborative-filtering similarity threshold $\tau$ moves from v1 default $0.7$ to v2 best $0.899$, with CV AUC moving from $0.682$ to $0.800$ --- $\Delta \approx +0.118$, a substantive improvement indicating that the v1 default was too permissive. The best $\tau$ sits at the upper boundary of the $[0.4, 0.9]$ search range, so a finer search may yield further improvement; this is noted as future work in Section~\ref{sec:eval-discussion}. Figure~\ref{fig:eval-hyperparams-optuna} shows the four panels (search landscape and TPE convergence trajectory for each parameter); the $\tau$-at-boundary and the $K$-flat patterns are both visually self-evident. The four BKT priors and the BKT mastery threshold were originally on the grid but were dropped from the v2 study as each trial would have required around thirty minutes of BKT refits; they remain at \texttt{config.py} defaults. Table~\ref{tab:eval-hyperparams} reports the headline numbers.
