## §4.9.0 — Preamble + parity table

**Cover (1 paragraph):**
- Instructor functionality implemented twice — V1 in `code/learning_dashboard/ui/views.py` (Streamlit, 8501); V2 in `code2/frontend/src/views/` (React, served by FastAPI on 8000).
- Both consume the same `learning_dashboard.*` analytics through different transport layers: V1 imports modules directly into the Streamlit process; V2 calls them over HTTP via FastAPI routers.
- Parity is intentional — V2 reimplements every V1 view for an apples-to-apples Chapter 5 comparison — and V2 adds one extra view (Session Progression) with no V1 equivalent.

## §4.9.1 — In-Class View

**Cover:**
- Two leaderboards side-by-side: top strugglers by descending struggle score, top hardest questions by descending difficulty, each capped at `LEADERBOARD_MAX_ITEMS = 15`.
- Bar colours follow threshold bands from §`sec:analytics` (green/yellow/orange/red mapped to On Track / Minor Issues / Struggling / Needs Help; same for difficulty).
- CF diagnostic panel below the leaderboards lists peers similar to flagged strugglers who haven't themselves crossed the threshold (§`sec:analytics`).
- Clicking a bar drills into Student Detail or Question Detail; V1 uses `st.plotly_chart(on_select="rerun")` + `point_index`; V2 routes via React Router to `/student/:id` or `/question/:id`.

## §4.9.6 — Settings View

**Cover (main paragraph):**
- Six toggles: Sound Effects (`SOUNDS_ENABLED_DEFAULT`), Auto-Refresh (`AUTO_REFRESH_DEFAULT`), CF (`cf_enabled`), Temporal Smoothing (`SMOOTHING_ENABLED`), Improved Models (`improved_models_enabled`), RAG (`RAG_ENABLED_DEFAULT`).

## §4.9.8 — Session Progression View (V2-only)

**Cover:**
- V2-only view in [`code2/frontend/src/views/SessionProgression.tsx`](../../../../code2/frontend/src/views/SessionProgression.tsx). Replays an ended session as 20 evenly-sized time buckets, animating struggle and difficulty leaderboards through the timeline.
- Useful for retrospective debriefs: when in the lab did a student deteriorate, or a question start attracting wrong answers, rather than only the final state.
- V1 has no equivalent — the bucket animation would require either rerunning the page at 50 ms cadence (not viable under Streamlit) or significant manual re-architecture; V2 handles it natively in React.

## Writing-style notes (match the user's voice in §4.8)

- Long sentences joined by commas/semicolons; 2–3 clauses per sentence is fine.
- "as" and "due to" for causal connections ("This is due to it being..."), not "because ..." / em-dash.
- "this" / "which" mid-sentence for elaboration.
- British spelling: "behavioural", "normalised", "whilst", "colour" (user mixes occasionally — don't over-correct).
- Light em-dash use; commas or semicolons preferred.
- "tradeoff" not "trade-off".
- The user's prose has some run-on tendency and a few typos slip through — don't try to flatten the voice; mirror it.
