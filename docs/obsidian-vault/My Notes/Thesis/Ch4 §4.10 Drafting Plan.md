### Key behaviours confirmed in both versions

- **5-second polling** for state changes. V1 via `st_autorefresh(interval=5000)` (line 460); V2 via `useApiData<LabState>('/lab/state', 5000)` (line 43).
- **6-character session code** from `LAB_CODE_LENGTH = 6`, validated client-side before POSTing.
- **Eligibility filter** for the waiting-state student list is identical: only `Struggling` and `Needs Help` levels are eligible (V1 line 254; V2 line 23 `URGENT_LEVELS`).
- **Self-claim** is gated by the instructor's `allow_self_allocation` flag (V1 line 264; V2 line 311). When disabled, only the instructor can dispatch.
- **Sound on assignment** — V1 `sound.play_assignment_received()` on None→assigned transition (line 491); V2's `useAssignmentSound` hook does the same gated by `settings.runtime.sounds_enabled`.
- **Two actions on assigned view**: Mark Helped (sets `status="helped"` but stays assigned, shows a "marked as helped" badge) and Release Student (actually unassigns). Same in both versions.
- **Shared state** via `data/lab_session.json` with file-lock — consistent with §4.5 Session Management.

### V1 ↔ V2 divergences worth flagging in prose

1. **Identity persistence**: V1 uses the URL `?aid=` query parameter (a side-benefit: a shared link can auto-rejoin). V2 uses `localStorage` via [`mobile/state/aid.ts`](../../../../code2/frontend/src/mobile/state/aid.ts) `readAid()`/`writeAid()`, keeping the URL clean.
2. **Stale-cache retry on join**: V2 wraps the first `/lab/join` POST in a transient-failure retry — if the cached lab state is stale (instructor just started the session), the mobile refetches and retries silently ([MobileApp.tsx:144-151](../../../../code2/frontend/src/mobile/MobileApp.tsx)). V1 has no equivalent short-circuit.
3. **Instructor dispatch console**: V2's `LabAssistantView` is a dedicated desktop view with a Dispatch Queue (top-10 ranked strugglers + auto-assign to first-idle button), Lab Assistants roster (with Release / Remove controls), and Active Assignments table. V1 exposes the same operations through the instructor sidebar widgets — different surface, same logic.

### Ch3 ↔ code discrepancy

**Ch3 §3.6 line 957 says**: "A *Mark Helped* action returns the assistant to the unassigned state."

**Code reality (both V1 and V2)**: Mark Helped only sets `status="helped"` and keeps the student assigned (V1 line 438-440; V2 `doMarkHelped` line 200-214). To unassign, the assistant clicks **Release Student** (V1 line 442-444; V2 `doRelease` line 216-228).

The §4.10 prose below describes the code accurately (two actions, not one). If you want Ch3 brought in line, that's a one-sentence edit at [`design-and-architecture.tex:957`](../../../../Report/main-sections/design-and-architecture.tex) separately.

### Constants confirmed

- `LAB_CODE_LENGTH = 6` in both [`code/learning_dashboard/config.py`](../../../../code/learning_dashboard/config.py) and [`code2/backend/config.py`](../../../../code2/backend/config.py).
- `RAG_MIN_SUBMISSIONS = 2` — assistant RAG panel short-circuits below this threshold (V1 line 377; V2 RAG path gates similarly).
- Auto-refresh interval hard-coded at 5000 ms in V1; V2 also hard-codes 5000 ms for `/lab/state` and 10000 ms for `/struggle`.

## Prose blocks (ready to paste)

### §4.10 preamble (paste at line 1280)

```latex
The lab assistant system is the runtime companion to the instructor dashboard described in Section \ref{sec:views}: a mobile-first surface that joined assistants use to receive student assignments during a live lab session, coordinated through the shared session file introduced in Section \ref{sec:session}. V1 ships the assistant as a separate Streamlit app served on port 8502 (\texttt{code/lab\_app.py}, delegating into \texttt{code/learning\_dashboard/assistant\_app.py}), whilst V2 ships it as a React mobile route mounted under \texttt{/mobile} on the same FastAPI process that serves the instructor SPA, with the four state views split across \texttt{code2/frontend/src/mobile/views/}. Both versions reach the same lab-state file through \texttt{learning\_dashboard.lab\_state} with a \texttt{filelock}, so an instructor running V1 and an assistant running V2 (or vice versa) see one another's state changes within the 5-second polling window. V2 also adds an instructor-facing dispatch console at \texttt{code2/frontend/src/views/LabAssistantView.tsx} which exposes the same allocation operations as V1's instructor sidebar but in a dedicated, full-width view.
```

### §4.10.1 Session Join Flow (paste at line 1286)

```latex
On first load, both versions present a single-screen join form: the assistant enters their name and the six-character session code (\texttt{LAB\_CODE\_LENGTH = 6}) shown on the instructor dashboard, and the client POSTs to the shared lab-state file via \texttt{learning\_dashboard.lab\_state.join\_session} (V1) or \texttt{POST /api/lab/join} (V2), receiving back an opaque assistant ID. Identity is then persisted so that a phone-browser refresh does not force a re-join, with V1 writing the assistant ID into the URL as the \texttt{?aid=} query parameter (which has the side-benefit that a shared link can auto-rejoin) and V2 writing it to \texttt{localStorage} via the small \texttt{readAid}/\texttt{writeAid} helpers in \texttt{mobile/state/aid.ts} so that the URL stays clean. V2 additionally wraps the first \texttt{POST /api/lab/join} in a transient-failure retry: if the mobile's cached lab-state is stale due to the instructor having just started the session, the failed join refetches \texttt{/api/lab/state} and retries silently rather than surfacing the error to the assistant. Once joined, the assistant remains enrolled until they tap \emph{Leave Session}, the instructor removes them from the roster, or the session ends.
```

### §4.10.2 Waiting and Assignment States (paste at line 1301)

```latex
Between assignments, the assistant lands on a waiting view that shows their name and the active session code at the top, alongside a ranked list of students who currently need help. The eligibility filter is identical in both versions: only students whose struggle band is \emph{Struggling} or \emph{Needs Help} appear in the list, ordered by descending struggle score and excluding any student already claimed by another assistant, so two assistants never end up at the same desk. The same list is visible to every joined assistant and to the instructor in real time as the lab-state file is the single source of truth and is polled every five seconds. When the instructor has enabled the \texttt{allow\_self\_allocation} flag in their dispatch controls, each row carries a \emph{Help \{student\}} button that lets the assistant self-claim from the list; when the flag is off, the row is read-only and the assistant waits for the instructor to dispatch them. Both behaviours coexist as the instructor sometimes prefers fully manual triage and sometimes wants to let experienced assistants self-organise.
```

### §4.10.3 Live Assistant Allocation and Mark-Helped (paste at line 1316)

```latex
Once a student is assigned, whether by the instructor or through a self-claim, the assistant view collapses into a single student card that carries the information the assistant needs at the desk: the student's anonymised ID, current struggle score and band-colour pill, a short list of \emph{Suggested Focus Areas} from the RAG layer described in Section \ref{sec:rag}, and the student's top three struggling questions ranked by mean incorrectness. The RAG panel short-circuits when the student has fewer than \texttt{RAG\_MIN\_SUBMISSIONS = 2} submissions in the current session as the suggestion would not be informative; in that case a small caption explains the wait. Two actions sit at the bottom of the card: \emph{Mark Helped} sets the assignment status to \texttt{"helped"} and shows a confirmation badge but leaves the assistant attached to the student, which lets the assistant keep checking in without releasing the slot, whilst \emph{Release Student} actually unassigns the student and returns the assistant to the waiting view so a fresh student can be dispatched. On the instructor side, V1 surfaces the equivalent allocation controls through dropdowns in the instructor sidebar whilst V2 adds a dedicated dispatch console (\texttt{code2/frontend/src/views/LabAssistantView.tsx}) that combines a roster of joined assistants, a top-ten Dispatch Queue of unassigned strugglers ordered by descending score, and a one-tap \emph{Dispatch} action that assigns the selected student to the first idle assistant.
```
