## Corrections to the Ch4 Rewrite Brief (verified against live code)

The base brief had four claims that disagreed with the live code. Use the corrections below instead of the originals:

1. **The model does not return a JSON object with a `bullets` field.** The prompt asks for "a JSON array of 2 or 3 short bullet strings (max 15 words each). No prose outside the array." The parser is permissive (accepts a bare list or any list-valued field), but the canonical contract is a JSON array of strings, not an object with a `bullets` key.
2. **Bullet length is "max 15 words", not "1–2 sentences".** The brief overstates the per-bullet budget.
3. **Cache invalidation is by session change (V1) or TTL (V2), not by submission-count threshold.** The brief's claim about a submission-count threshold is not in the code. V1: plain dict `_suggestion_cache[student_id]`, cleared on session change. V2: `TTLCache(maxsize=256, ttl=600)` keyed by `(student_id, window.from_, window.to_, window.module, session_id)`.
4. **No student identifiers are sent to OpenAI.** The prompt carries only struggle score, struggle label, top-3 questions and incorrectness rates, and the Q&A snippets. The `student_id` is used locally for the pandas filter and as a cache key — it never leaves the process. Stronger privacy story than the brief gave it.

Plus one structural surprise: **RAG has two entry points, not one.** `generate_student_suggestions()` (the assistant-panel one) and `generate_question_suggestions()` (Question Detail). Same two-layer retrieval, same generator; only the Layer-1 pandas filter differs.

## §4.8.1 — Hybrid Two-Layer Architecture

**Maths recap (optional):** Layer 2 is top-$k$ nearest-neighbour in a 384-D embedding space using cosine similarity through ChromaDB's HNSW index.

**Single paragraph (cover):**
- Layer 1: pandas pre-filter `df[df["user"] == student_id]` — narrows to the student's own history.
- Layer 2: ChromaDB semantic search over those embeddings, with `where` clause keyed by `session_{session_id}` collection.
- Output: (submission text, feedback, score) triples handed to the generator.
- V2 wraps the synchronous build+generate path in `asyncio.to_thread` so the Chroma build doesn't block the FastAPI event loop. V1 doesn't need this — the build happens during the cached Streamlit scoring pass.
- Both `generate_student_suggestions` and `generate_question_suggestions` use the same two-layer pattern; only the Layer-1 filter differs.

**Voice/length:** 1 paragraph + optional maths recap.

**Citations:** `\cite{lewisRetrievalAugmentedGenerationKnowledgeIntensive2020}` (already in `references.bib`).

## §4.8.2 — Embedding Model (all-MiniLM-L6-v2)

**Single paragraph (cover):**
- `sentence-transformers` running `all-MiniLM-L6-v2` locally — no API call. 384-D, ~80 MB, distilled from BERT-base.
- ChromaDB collection `session_{session_id}`; persistence at `data/rag_chroma/` via `PersistentClient`.
- `RAG_MIN_SUBMISSIONS = 2` short-circuits the panel for students with <2 submissions.
- `RAG_ENABLED_DEFAULT = True` but missing optional deps silently no-op the path (see §4.8.5).

## §4.8.3 — Generation and Prompting

**Single paragraph (cover):**
- Generator is gpt-4o-mini (reuses the §4.6.1 dependency surface).
- Prompt is system message (terse, practical, assistant-facing tone) + user message containing the retrieved evidence (Figure `fig:code-ragprompt`).
- Output constrained to JSON via `response_format={"type": "json_object"}`; prompt asks for a JSON array of 2–3 bullet strings, max 15 words each.
- Each bullet addressed to the assistant (not the student) and frames a specific thing to look at.

**Figure (`fig:code-ragprompt`):** verbatim from `rag.py:263-273` — the system + user message construction. Caption notes the privacy property (no student id in the prompt).

## §4.8.4 — Suggestion Caching

**Single paragraph (cover):**
- V1: plain dict `_suggestion_cache[student_id]`, cleared on session change.
- V2: `TTLCache(maxsize=256, ttl=600)` keyed by `(student_id, window.from_, window.to_, window.module, session_id)`.
- Neither invalidates on a submission-count threshold; freshness governed by session boundary (V1) or 10-minute TTL (V2). Deliberate trade-off: RAG is the expensive step; re-running on every poll would dominate OpenAI cost.

## §4.8.5 — Graceful Degradation and Privacy

**Single paragraph (cover):**
- `_lazy_import()` guards `chromadb` and `sentence-transformers`; missing either → entire RAG path silently no-ops, rest of assistant UI unaffected. `RAG_ENABLED_DEFAULT` honoured but dep check has final word.
- OpenAI failure/timeout → empty list returned, cached bullets remain on screen, no error banner. V2 sets explicit `timeout=15.0`.
- Privacy: prompt contains struggle score, struggle label, top-3 questions + incorrectness rates, and Q&A snippets. **No user identifier of any kind in the prompt.** `student_id` is local pandas-filter / cache-key only.

**Voice/length:** 1 paragraph.
