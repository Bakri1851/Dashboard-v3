## §4.6.1 — Incorrectness Scoring (OpenAI, batched, cached)

**Maths recap:** Each submission $i$ receives a scalar incorrectness $x_i \in [0,1]$ where $x_i = 0$ means "fully correct" and $x_i = 1$ means "fully incorrect"; intermediate values capture partial credit. The mapping from the `ai_feedback` text to $x_i$ is delegated to gpt-4o-mini (`OPENAI_MODEL`); empty feedback returns the neutral fallback $x_i = 0.5$ without an API call.

**Paragraph 1 — how it works:**
- Calls go through `_call_openai_batch()`; up to `OPENAI_BATCH_SIZE = 20` **feedback texts** per request — smaller batches parse more reliably than the API's nominal max. (Not "(question, answer, feedback) triples" — the prompt only carries the feedback strings.)
- Cold-start guard `SCORING_PER_RUN_CAP = 500` caps the new pairs scored per Streamlit rerun (V1) or per FastAPI cache rebuild (V2); the rest fall back to 0.5 and are scored on subsequent runs.
- On API failure the row falls back to 0.5 and the bad response is **not** persisted — next rerun retries.

**Paragraph 2 — caching, V1 vs V2:**
- Both versions keep the in-process `_incorrectness_cache` keyed by feedback text (introduced in §4.4.3).
- V2 adds a disk-persisted JSON at `data/incorrectness_cache.json` whose payload is **model-versioned**: `OPENAI_MODEL` name stored alongside; cache discarded on load when model name changes.
- V2's `compute_incorrectness_column` exposes two kwargs used by the lifespan handler: `max_new_scores` overrides the per-run cap; `score_new=False` is a lookup-only mode (no OpenAI calls). The warm-ratio short-circuit in `fig:code-warmratio` sets `score_new=False`.

**Figure:** `fig:code-openai` — lstlisting of the prompt + API call inside `_call_openai_batch()` (about 17 lines from `analytics.py:82-102`).

**Voice/length:** Maths recap + 2 paragraphs + 1 figure.

## §4.6.2 — Baseline Student Struggle Model (7 signals)

**Maths recap:**

$$S(s) = \alpha N_s + \beta T_s + \gamma I_s + \delta R_s + \eta A_s + \zeta D_s + \theta\,\mathrm{REP}_s$$

where each signal is min-max normalised across the cohort. Bayesian shrinkage:

$$w_n = \frac{n}{n + K}, \qquad S_{\text{final}} = w_n S + (1 - w_n)\bar{S}, \qquad K = 5.$$

Recent-incorrectness $A$ uses exponential time decay $w_i = e^{-\lambda \Delta t}$ with half-life $T_{1/2} = 1800$ s, giving $\lambda = \ln 2 / 1800$.

**Paragraph 1 — anchor the maths recap to the seven signals + describe the implementation pattern:**
- Open by binding the formula above to Table~\ref{tab:struggle-7sig}: the weighted sum runs over seven cohort-normalised signals; the seven weights live in `config.py` and an import-time assertion enforces sum-to-1.
- Implementation pattern is **per-student groupby + DataFrame-level vectorisation**, not a single vectorised pass. The `for user, group in grouped:` loop computes the raw signals (`n_raw`, `t_raw`, `i_hat`, `r_hat`, `A_raw`, `d_raw`, `rep_hat`) into a row dict; the assembled DataFrame is then min-max normalised column-wise and weighted-summed without a second loop.
- Bayesian shrinkage closes the pass via $w_n = n / (n + K)$ with $K = 5$: a student with $n \ll K$ is pulled almost entirely toward the cohort mean $\bar{S}$, while a student with $n \gg K$ keeps their raw score — this is what "low-$n$ students pulled toward the cohort mean" means operationally.

**Paragraph 2 — V1/V2 divergence + threshold bands:**
- V1 takes the unweighted mean of `incorrectness` for both $I$ (overall) and $A$ (recent). V2 weights **both** by per-row measurement confidence (forward ref: Section~\ref{sec:advanced}) — low-confidence rows (empty/short feedback, mid-range LLM scores) contribute less mass without being dropped, so $I$ and $A$ never collapse to 0 on all-empty cohorts.
- Final classification uses `STRUGGLE_THRESHOLDS`: On Track (0.00–0.20), Minor Issues (0.20–0.35), Struggling (0.35–0.50), Needs Help (0.50–1.00). The bands feed both leaderboard colouring and the In-Class view's risk grouping (Section~\ref{sec:views}).

## §4.6.4 — Collaborative Filtering

**Maths recap:** For each pair of students $(s, t)$, cosine similarity over a five-element behavioural feature vector $\mathbf{v} = (N, T, I, R, A)$:

$$\mathrm{sim}(s, t) = \frac{\mathbf{v}_s \cdot \mathbf{v}_t}{\lVert \mathbf{v}_s \rVert\, \lVert \mathbf{v}_t \rVert}.$$

A struggling student $s^\star$ "infects" any peer $t$ with $\mathrm{sim}(s^\star, t) \geq \tau$ and $t$ not already flagged.

**Single paragraph:**
- Five-element `CF_FEATURES`: four normalised baseline signals (`n_hat`, `t_hat`, `i_norm`, `A_norm`) plus `d_hat`.
- $k = 3$ nearest neighbours per flagged struggler.
- Threshold τ default: **V1 = 0.6, V2 = 0.7** — exposed as a slider in the Settings view (§4.9.6).
- Output: list of (similar student, source student, similarity) tuples in the In-Class view's CF diagnostic panel.
- Toggle `cf_enabled` in session state hides the panel when off.

**Voice/length:** Maths recap + 1 paragraph.

## §4.6.5 — Mistake Clustering

**Single paragraph:**
- For each question with at least `CLUSTER_MIN_WRONG = 3` incorrect submissions, `cluster_question_mistakes()` TF-IDF–vectorises the wrong-answer texts and clusters them with KMeans.
- $k$ selected by silhouette score, capped at `CLUSTER_MAX_K = 5`. Each cluster is labelled by a short OpenAI call (`_label_clusters_with_openai`).
- Up to `CLUSTER_MAX_EXAMPLES = 3` representative answers per cluster, truncated to `CLUSTER_EXAMPLE_MAX_CHARS = 300`.
- Cache key uses SHA1 of input texts, not Python's built-in `hash()` (which is salted per process and would invalidate every restart).
- Results surface in the Question Detail view (§4.9.3).

**Voice/length:** 1 paragraph.
