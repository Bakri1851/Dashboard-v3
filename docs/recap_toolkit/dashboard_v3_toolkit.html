Report guide

Dashboard V3 is a live learning-analytics system for university lab sessions. Two Streamlit apps — an instructor dashboard and a mobile lab-assistant app — poll a shared API, score student struggle and question difficulty in real time, and coordinate instructor-to-assistant handoffs through a file-locked JSON state store. This toolkit is a single-file tour of how every piece works.

What it does

Watches a live cohort during a lab. Flags students who need help using a 7-signal composite score, surfaces which questions are hard, and hands struggling students off to lab assistants for 1-to-1 coaching — augmented with AI-generated coaching bullets.

Who uses it

🖥️
InstructorDesktop dashboard on port 8501. Starts the session, watches leaderboards, assigns assistants, closes the session.
📱
Lab assistantMobile app on port 8502. Joins via 6-char code, gets paged with an assigned student, receives RAG coaching bullets, marks "helped".

Tech stack

Streamlit · pandas · NumPy · SciPy (L-BFGS-B) · scikit-learn (cosine sim, TF-IDF, K-means) · ChromaDB · OpenAI GPT-4o-mini · sentence-transformers (all-MiniLM-L6-v2) · filelock. Everything runs locally — no server needed.

Where state lives

data/lab_session.json (shared session + assignments, file-locked) · data/saved_sessions.json (replayable session snapshots) · data/rag_chroma/ (ChromaDB embeddings) · st.session_state (in-process UI state) · process-local caches (_incorrectness_cache, _df hash).

Alternative frontend · code2/

Additive second frontend — React + Vite + TypeScript served by a FastAPI backend on port 8000. Built in a code2/ shadow copy so code/ stays untouched (defence-day fallback). Same data/lab_session.json + FileLock coordinates all four processes. 7 swappable themes (paper / newsprint / solar / scifi / blueprint / matrix / cyberpunk) matching the editorial mockup. End-to-end flow (live session)

Student submits an answer to the AI tutor platform — outside this system. The AI returns feedback text.
Dashboard polls the API via fetch_raw_data() every 10s (Streamlit cache TTL). Newline-delimited JSON with embedded XML per submission.
Parse + normalise → DataFrame. Drop test modules, parse timestamps, tag academic period. Cached by SHA1 hash of the raw response.
Incorrectness scoring. Unique feedback strings batch-sent to GPT-4o-mini — returns [0,1] floats. Cached per process.
7 struggle signals computed per student, cohort-normalised, weighted-summed, Bayesian-shrunk toward class mean. Output: S_final and a label pill.
Leaderboard renders. Instructor spots a struggling student, clicks the bar → detail view with RAG-generated coaching bullets.
Instructor assigns a lab assistant from a dropdown. Write lands in lab_session.json under filelock.
Assistant's phone picks up the assignment on its next 5s auto-refresh. Sees the student's detail card and coaching bullets.
Assistant taps "Mark as Helped" → assignment cleared → back to waiting view. Instructor sees the assistant free again.

Case study — Alice's journey through the pipeline

Scenario: Alice submits an off-track answer during minute 18 of a 50-minute lab.
Submission→ AI feedback: "Your integral bounds are swapped…"
GPT-4o-mini→ incorrectness = 0.72
Add to A_raw (last 5, 30-min decay)→ A_raw = 0.68
Cohort-normalise 7 signals→ A_norm=0.81, i_norm=0.65, r_norm=0.40, …
Weighted sum→ S_raw = 0.58
Shrinkage (n=7, K=5)→ w_n = 7/12 = 0.58 · S_final = 0.58·0.58 + 0.42·0.40 = 0.50
Classify→ 0.50 → "Needs Help" (red pill)
Instructor clicks Alice→ RAG queries ChromaDB for her worst question, ships 3 retrieved snippets + scores to GPT-4o-mini
GPT returns→ 3 coaching bullets ("check bound order", "re-read setup", "try substitution")
Assign to "Ben" (lab assistant)→ lab_session.json updated under filelock
Ben's phone refreshes (5s)→ sees Alice's detail card + the 3 bullets → walks over
Ben taps "Mark Helped"→ assignment cleared, back to waiting view, instructor sees Ben free

Everything starts here. The external AI tutoring platform records student submissions and generates feedback. Your dashboard polls that data, parses it, cleans it, and builds a DataFrame that every other system reads from.

1
Student submits an answer (external system)
The AI tutoring platform stores: the student's answer (student_answer), AI feedback text (ai_feedback), question ID, module code, session ID, user ID, and timestamp. Your code never sees this happen in real time — you only see it when you poll.
2
fetch_raw_data() — GET request to the API
Decorated with @st.cache_data(ttl=10). Streamlit stores the response and reuses it for 10 seconds. If the app reruns within 10s, it returns the cached string without touching the network. After 10s the cache expires. Returns raw text — either JSON or XML.
3
detect_format(raw) — JSON or XML?
Takes the first line, tries json.loads(). Success → JSON path. JSONDecodeError → XML path.
4
parse_json_response(raw) — the tricky one
Newline-delimited JSON — one object per line. Each object has top-level fields PLUS an xml field containing an embedded XML string. Wraps it in <root>, parses with xml.etree, walks .//submission elements — each has <srep> (answer) and <feedback><response> (AI feedback). One submission element → one record dict. Falls back gracefully on parse errors.
5
normalize_and_clean(records) — build the DataFrame
1. Convert to DataFrame. 2. Drop rows in EXCLUDED_MODULES (AI_TEST, 24COB231, 24WSC701). 3. Apply MODULE_RENAME_MAP. 4. Parse timestamps — invalid → NaT → dropped. 5. Sort ascending. 6. Add academic_period column.
6
load_data() — double caching
Calls fetch → detect → parse → clean. Second caching layer: hashes the raw response string and stores the DataFrame in st.session_state["_df"]. If hash matches on next rerun, skips parse+clean entirely. Returns (DataFrame, error_msg) on any error returns empty DataFrame with correct schema.

Every downstream signal depends on incorrectness — a continuous [0,1] score of how wrong a submission is. An LLM (gpt-4o-mini) judges ai_feedback text and returns the number. This is LLM-as-judge methodology: use a language model as the scorer when no labelled ground truth is available.

1
compute_incorrectness_column(df) — gather candidates
Scan the DataFrame's ai_feedback column. Skip rows already in _incorrectness_cache. Skip empty/whitespace feedback (those get 0.5 fallback without an API call). Collect unique non-empty strings.
2
Batch to GPT-4o-mini
Chunk to batches of 50. Per batch: one system prompt + user prompt with numbered feedback list. Model returns a JSON array of floats in [0,1]. Response parsed, cached by feedback hash.
3
Failure modes — safe defaults
empty feedback → 0.5 (no API call)
OpenAI parse failure → 0.5 × batch_size
OpenAI timeout → 0.5 × batch_size
0.5 is deliberate: it is the most "uncertain" point. A 0.5 fallback treats the submission as neither clearly correct nor clearly wrong, so it contributes equal weight to both sides of the CORRECT_THRESHOLD cliff. Downstream code treats a flood of 0.5s as a signal that scoring has failed (BKT/IRT refuse to fit on single-class data for this reason).
4
Why LLM-as-judge is defensible here
Labelled ground truth (instructor-graded correctness) does not exist for this dataset. The LLM replaces a rubric-based human grader. This is the Zheng et al. (2023) methodology from Judging LLM-as-a-Judge with MT-Bench. Known calibration risks: the model can be overconfident at the extremes; the 0.5 fallback masks silent failures; the score is not interpretable per-token. These are documented limitations — not bugs.
Thesis Ch5 §5 reliability subsection: report the fraction of rows at 0.5 fallback. A >80% fallback rate indicates OpenAI is failing silently or ai_feedback is mostly empty — check .streamlit/secrets.toml and data-pipeline parsing.

The core of your project. All 7 signals are cohort-normalised before the weighted sum so configured weights match effective weights.

1
compute_incorrectness_column(df) — the foundation signal
Collects unique non-empty ai_feedback strings not in _incorrectness_cache. Sends to GPT-4o-mini in batches of 50. Returns JSON array of [0,1] floats. Empty feedback → 0.5 without API call. OpenAI failure → [0.5]*batch_size.
2
The 7 signals — all cohort-normalised
n → n̂ (10%)
Submission count. Min-max normalised across all students.
t → t̂ (10%)
Time active in minutes. Min-max normalised.
î → i_norm (20%)
Mean incorrectness across all submissions. Cohort-normalised so weight means what it says.
r̂ → r_norm (10%)
Retry rate = 1 − (unique_questions / n). Cohort-normalised.
A_raw → A_norm (38%)
Heaviest weight. Exponentially time-decayed mean of last 5 submissions. 30-min half-life. Cohort-normalised before formula.
d → d̂ (5%)
np.polyfit slope of incorrectness vs submission order. Positive = getting worse. Min-max normalised.
rep̂ → rep_norm (7%)
Exact-answer repetition rate. Cohort-normalised.
3
Formula and Bayesian shrinkage
S = 0.10·n̂ + 0.10·t̂ + 0.20·i_norm + 0.10·r_norm + 0.38·A_norm + 0.05·d̂ + 0.07·rep_norm
min_max_normalize() returns 0.5 (not 0.0) when all values are equal — preserves weight contribution rather than silently zeroing it.
w_n = n / (n + 5)
S_final = w_n × S + (1 − w_n) × class_mean
Shrinkage is the final transform. n=2: 71% pulled toward mean. n=10: 33%. n=50: barely affected.
4
EWMA temporal smoothing (distinct from A_raw decay)
S_smooth(t) = α · S(t) + (1 − α) · S_smooth(t − Δt) with α = 0.3
Enabled by config.SMOOTHING_ENABLED=True with SMOOTHING_ALPHA=0.3. Applied across snapshot refreshes — damps second-by-second jitter when the dashboard auto-reruns. This is a different technique from A_raw's per-submission exponential time decay (Ebbinghaus), which weights individual submissions by their age. Two time scales: A_raw lives inside a single score computation (older submissions contribute less); EWMA lives across score computations (this minute's score blends with last minute's).

The Rasch model
P(correct | θ_i, b_j) = sigmoid(θ_i − b_j) = 1 / (1 + exp(−(θ_i − b_j)))
θ_i = student ability. b_j = question difficulty. Student succeeds when ability exceeds difficulty.

BKT models learning as a Hidden Markov Model. Hidden state = whether the student knows the skill. Observed = correct/wrong. Bayes' rule updates the belief after each submission.

ParameterSymbolDefaultMeaning

Defaults are textbook values (Corbett & Anderson). Settings → Model Toggle exposes a Fit from data action that replaces them with MLE estimates from the current filtered view.
1
bkt_update() — the core equation
If correct: P(L|obs) = P(L)·(1−P(S)) / [ P(L)·(1−P(S)) + (1−P(L))·P(G) ]
If wrong: P(L|obs) = P(L)·P(S) / [ P(L)·P(S) + (1−P(L))·(1−P(G)) ]
Transition: P(L_next) = P(L|obs) + (1 − P(L|obs)) × P(T)
Runs once per submission per question per student. Result clipped to [0,1].
2
compute_all_mastery() — replay with deterministic sort
BKT is a sequential HMM — replay order determines the posterior. Explicitly sorts by [timestamp, question] using mergesort before grouping. Secondary key on question ensures deterministic output when submissions share identical timestamps (bulk imports).
3
fit_bkt_parameters() — MLE from submission history
L(θ) = Σseq Σt log P(obst | obs1..t−1, θ)
θ* = argmaxθ L(θ), θ = (P(L₀), P(T), P(G), P(S))
Forward-algorithm log-likelihood of the 2-state BKT HMM, optimised with L-BFGS-B. Bounds [(0,1), (0,1), (0,0.5), (0,0.5)] enforce the identifiability constraint P(G)+P(S)<1. Same [timestamp, question] ordering as inference, so the fit sees the exact sequences compute_all_mastery() will replay. Three-component model: behavioral + BKT mastery gap + IRT difficulty-adjusted. Is the student performing below what they should know? Are they failing easy questions?

1
Three components with graceful degradation
S_improved = 0.45·behavioral + 0.30·mastery_gap + 0.25·difficulty_adjusted
BKT unavailable → w_mg to w_beh (behavioral=0.75). IRT unavailable → w_da to w_beh (behavioral=0.70). Coverage guard: if <50% of students have mastery records, w_mg redistributed entirely. Weight-sum invariant asserted at runtime.
2
Component 1: behavioral composite (45%)
behavioral = (A_norm + r_norm + d̂ + rep_norm) / 4
All sub-signals cohort-normalised before averaging so equal weighting is genuinely equal.
3
Component 2: mastery gap (30%)
recent_performance = 1 − A_raw
mastery_gap = max(0, mean_mastery − recent_performance)
How far below demonstrated knowledge is the student performing right now? Missing mastery imputed to class_mean (not 0.0) — unknown should read as average, not zero.
4
Component 3: difficulty-adjusted incorrectness (25%)
difficulty_adjusted = coverage·user_mean + (1−coverage)·cohort_mean
Coverage = n_covered / n_recent (last 5 subs). A student with 1-of-5 covered leans 80% on the cohort mean. Failing easy questions (low IRT difficulty) scores higher than failing hard ones.
5
Post-assembly: Bayesian shrinkage only
Bayesian shrinkage is the final transform. The previous min-max normalisation step was removed — it was fighting the shrinkage (shrinkage pulls low-n students toward the mean; normalisation spread them back out).
6
Three design mechanics to cite in Ch3 §3.4.4
Coverage-weighted shrinkage
In _compute_difficulty_adjusted: score = coverage·user_mean + (1−coverage)·cohort_mean. Distinct from n/(n+K) classical shrinkage — the weight is data coverage, not sample size. Low IRT-coverage students lean on the cohort mean.
Mean imputation of missing mastery
mean_mastery absent → imputed to class mean, not 0.0. "Unknown" should not read as "zero mastery" (which would silently zero the mastery-gap contribution). Cite Little & Rubin (2002).
Graceful-degradation redistribution
BKT unavailable → w_mg → w_beh (0.75/0.00/0.25). IRT unavailable → w_da → w_beh (0.70/0.30/0.00). Both → w_beh = 1.0. A runtime assert abs((w_beh + w_mg + w_da) - 1.0) < 1e-9 at improved_struggle.py:168 guarantees the invariant.

The model comparison view scatters baseline vs improved scores for the same students (and baseline vs IRT for the same questions), so you can see at a glance where the models disagree. Agreement is currently reported as a raw percentage; Cohen's κ is the supervisor-grade upgrade.

1
Scatter: x = baseline, y = alternative
Points on the diagonal → models agree on the score. Points above the diagonal → the alternative model rates higher. Points below → lower. The colour of each point = the alternative model's classification level pill, so you can see disagreement type (level flip vs score drift within the same level).
2
Agreement summary — raw %
agreement = |{i : baseline_level(i) == alternative_level(i)}| / N
Simple same-level count divided by total. Implemented in render_agreement_summary(). Intuitive but not chance-corrected: if both models always returned "Minor Issues" agreement would be 100% without either model being useful.
3
Cohen's κ — chance-corrected agreement (suggested upgrade)
κ = (p_o − p_e) / (1 − p_e)
p_o = observed agreement (raw %)
p_e = expected agreement under independent random classification
κ discounts agreement that would happen by chance alone. Guidelines: κ < 0.20 poor, 0.21–0.40 fair, 0.41–0.60 moderate, 0.61–0.80 substantial, >0.80 almost perfect. Not yet implemented in the comparison view.
4
Disagreement table — sorted by |Δ|
Below the scatter: a table of the N biggest score differences between the two models, for instructor review. Colour-coded by delta magnitude. This is where unusual cases (systematic disagreement, outliers) become visible.
5
Spearman ρ — rank concordance
ρ = Pearson corr(rank_baseline, rank_improved)
domain: ρ ∈ [−1, 1] · computed on the common student set (|common| ≥ 3)
Measures whether the two models order students the same way, independent of the score values. Complementary to raw agreement %: two models can disagree on level boundaries but still produce near-identical rankings (high ρ). Interpretive bands: >0.7 strong, 0.3–0.7 moderate, <0.3 weak. Returns null when fewer than 3 common entities or zero rank variance. Exposed as spearman_rho on /api/models/compare; implemented in code2/backend/routers/models_cmp.py:_spearman.
6

overlap = |top10_baseline ∩ top10_improved| / 10
Operational complement to Spearman ρ: when only the most-at-risk tail matters (instructor triage), top-10 overlap answers "do both models flag the same ten students". Exposed as top10_overlap on the same endpoint.

Why the code looks the way it does. Each answer is the real reason, not the textbook one.

QWhy 0.5 as the fallback incorrectness value, not 0.0 or random?▸
0.5 is the neutral midpoint — neither biasing toward "correct" nor "incorrect" — and preserves the feature's contribution in downstream weighted sums without introducing artificial signal. An empty feedback string is a missing measurement, not evidence of correctness. Same logic applies to min_max_normalize() returning 0.5 on a degenerate cohort.
QWhy a 30-minute half-life for recent incorrectness decay?▸
30 minutes balances recency (catching acute struggle within a lab session) with robustness to momentary lapses. Shorter half-lives over-weight single mistakes; longer ones miss real-time trends. Exposed in DECAY_HALFLIFE_SECONDS for tuning.
QWhy SHRINKAGE_K = 5 specifically?▸
Students with 1–2 submissions produce very noisy struggle scores. Shrinkage pulls them toward the class mean, reducing false positives. w_n = n/(n+5) means a student needs ~5 submissions before their individual score gets 50% weight — a pragmatic threshold matching typical lab-session engagement in the first 5–10 minutes.
QWhy three separate weights in the improved model — why not learn one combined metric?▸
Three interpretable signal groups let an instructor understand which dimension drove a score ("struggling on easy questions" vs "underperforming vs demonstrated mastery"). Learned weights would be black-box and harder to defend in a thesis. The fixed weights also prioritize behavioural signals (real-time) over model-derived signals (longitudinal) for session-scale responsiveness.
QWhy graceful degradation instead of requiring all models to be available?▸
Cold-start and missing-data are unavoidable in live sessions — IRT needs ≥2 distinct difficulty values, BKT needs observation history, improved-struggle needs ≥50% mastery coverage. If a prerequisite is missing, redistributing that component's weight to the remaining components keeps the model producing useful output. Instructor gets partial insight rather than none, and the weight-sum invariant still holds.
QWhy ChromaDB for RAG instead of simpler keyword search?▸
Semantic search via embeddings captures conceptual similarity (e.g. "I divided wrong" ↔ "arithmetic error") that keyword search misses. all-MiniLM-L6-v2 is local (no API cost or latency), compact (~90MB), and good enough for short feedback strings. ChromaDB adds metadata filtering so the two-layer architecture can enforce per-student privacy cheaply.
QWhy two-layer RAG (pandas filter + ChromaDB)?▸
Layer 1 (pandas) enforces privacy and identifies the student's worst question. Layer 2 (ChromaDB where filter + semantic query) finds the most relevant snippets within that student's own history only. Pure-semantic search without a structured pre-filter could surface another student's feedback. Architecture credited to Dr. Batmaz.
QWhy SHA1 in data_loader, not Python's built-in hash()?▸
Python's hash() is seeded randomly per-process for security (PYTHONHASHSEED). That means the same string gives a different hash across reruns, breaking any cache key that depends on cross-process reproducibility. SHA1 is deterministic and — for non-adversarial cache use — fine on performance.
QWhy a 6-char session code instead of URLs or login?▸
Codes are human-readable, shareable (instructor says the code out loud or writes it on the board), and decouple session identity from any web auth. Characters exclude O/0/I/1 to avoid confusion when read aloud. Prevents accidental cross-session state leakage on a shared disk.
QWhy disk-backed lab-session state (lab_session.json + filelock) instead of a server DB?▸
No server infrastructure needed — the entire system deploys as two streamlit run commands. Matches Streamlit's stateless architecture (UI state is disposable; shared state is on disk). Trade-off: correctness depends on the file-locking backend being reliable and on the 5s polling cadence — fine for tens of users, wouldn't scale to hundreds.
QWhy mastery_gap = max(0, mean_mastery − recent_performance) clamped at zero?▸
A negative gap means the student is currently outperforming their demonstrated mastery — they're on a hot streak. That's not a struggle signal. Without the clamp, "good day" students would actively reduce their struggle score, which is the wrong direction. Clamping to zero means the mastery-gap component can only add to struggle, never subtract.
QWhy cohort-normalise every signal before the weighted sum?▸
Configured weights have to match effective weights. i_hat, r_hat, A_raw, rep_hat are already in [0,1] but not on the same cohort-relative scale — A_raw=0.8 could be the top student in one cohort and the median in another. Without normalisation, the 38% weight on A_norm could contribute 60% or 15% of the final score depending on class distribution. Cohort normalisation makes the weights do what they say.
QWhy sort BKT observations by [timestamp, question] using mergesort?▸
BKT is a sequential HMM — the posterior depends on replay order. Identical timestamps happen during bulk imports. Mergesort is stable, so ties fall back to the secondary key (question), producing deterministic output every run. Without this, the same data could produce different mastery scores on consecutive reloads.

Defence prep — why this method?
Q1. An examiner asks: "Why is BKT appropriate here when DKT (Deep KT) has higher reported AUC in the literature?"
BKT has interpretable parameters (P(L₀), P(T), P(G), P(S)) and a closed-form Bayesian update. A lab instructor needs to know why mastery is 0.4 and what it would take to move it — not just a softmax output. DKT needs thousands of sequences to train; a single lab session has dozens to low hundreds of attempts per skill. BKT also fits in seconds via L-BFGS-B and degrades gracefully when a skill has few observations. Cite Corbett & Anderson (1995) for BKT; acknowledge Piech et al. (2015) DKT as the modern alternative in Ch6 Future Work.
Q2. "Your composite struggle score has seven signals with weights you set by hand. How is that not arbitrary?"
Three defences. (1) OECD/JRC (2008) composite-indicator framework: every signal is min-max-normalised across the cohort so weights reflect relative importance rather than scale artefacts; a weight-sum invariant is asserted at import time. (2) Bayesian shrinkage (Efron & Morris 1977) pulls low-n students toward the class mean, so noise on the first 3-5 submissions cannot flip a classification. (3) The weights are a documented limitation, not a hidden choice: Ch5 §5.5 states they were set by trial and error in the absence of labelled ground truth, and Ch6 proposes ML-based optimisation once labels exist.
Q3. "Why LLM-as-judge for incorrectness instead of a rule-based scorer?"
No labelled ground truth exists for this dataset. A rule-based scorer would need a hand-crafted rubric per question; the platform generates feedback text dynamically, so per-question rules do not transfer. Zheng et al. (2023) show LLM-as-judge agrees with human raters at ≥80% on reasoning tasks when prompted carefully. Limitations are named explicitly in Ch5: calibration is not perfect; 0.5 fallback masks silent failures; BKT and IRT both refuse to fit on single-class data as a tripwire for exactly this.
Q4. "Agreement % between baseline and improved struggle is 0% in your comparison view. Isn't that a bug?"
No — it is a known limitation documented in Ch5 §5.5. Improved struggle collapses everything to max struggle under sparse session data because (a) IRT difficulty is near-constant with few questions per student, so the difficulty-adjusted component saturates, and (b) BKT mastery sits near P(L₀) without enough opportunities to transition. Framed as a data-coverage problem, not a model failure: the dashboard handles this with graceful degradation (w_mg redistributed when coverage < 0.5), but the comparison view surfaces the structural disagreement honestly rather than hiding it.
Q5. "Your clustering uses silhouette to auto-pick k. Why not elbow method?"
Elbow method plots within-cluster inertia vs k and picks the visual "elbow" — subjective and unstable for small n. Silhouette (Rousseeuw 1987) is bounded in [-1, +1], has an explicit trade-off between cohesion and separation, and produces a scalar that is straightforward to argmax. For the mistake-clustering use case with 3–30 wrong answers per question, silhouette is more robust.

Section-by-section guide to writing the thesis. Each chapter below expands to show: what the chapter proves, how long it should be, the required subsections, which parts of this toolkit to draw from, and the pitfalls the example theses reveal. The structure is a hybrid of the two supplied reports that best match Dashboard V3 — Desktop SQL Editor (two-app system, dataset, evaluation) and NLP for Disaster Detection (algorithm breakdown, model comparison, accuracy tables).

Target total length: 40–60 pages for a typical Loughborough CS masters thesis (front + back matter excluded). Work chapters roughly in order, but draft the abstract and the conclusion last.

Writing principles
  • Past tense for what you did ("I implemented…"), present tense for what the system does ("The scoring engine normalises…").
  • Third-person or passive in the abstract and conclusion ("A live learning-analytics system was built…"). First-person ("I") is acceptable in Loughborough CS theses in reflection/implementation sections.
  • Every figure must have a number, caption, and be referenced in the prose ("Figure 4 shows…"). Don't drop unexplained figures.
  • Every formula must be explained in the line before or after. Don't dump LaTeX at the reader.
  • Every design choice must be justified — either by citation, by a requirement, or by an experiment you ran.
  • Consistency of notation — if the literature review calls it S, implementation and evaluation also call it S. Don't switch to struggle_score halfway through.
  • Cite the primary source, not Wikipedia. Use Google Scholar to find the original paper.

Chapter 2 — Literature Review & Background
~8–12 pagesTheory · Related systems · Gap analysis
▸

Purpose. The biggest chapter. Prove you know the field well enough to judge what has and hasn't been done, define the vocabulary the rest of the thesis uses, and end with a gap that specifically your project fills. Aim for ~25 references minimum.

Required subsections
  1. 2.1 Learning analytics — overview (~1 page). Definition, the real-time vs post-hoc distinction, why live matters for formative feedback. Cite Siemens & Long (Penetrating the Fog, 2011), Baker & Inventado (Educational Data Mining and Learning Analytics, 2014).
  2. 2.2 Student struggle & behavioural signals (~1–1½ pages). What behavioural signals have been correlated with struggle in the literature: submission volume, time-on-task, error patterns, retry rate, help-seeking, hint abuse. Reference Baker's "gaming the system" work, the EDM literature's struggle-detection tradition.
  3. 2.3 Knowledge tracing — Bayesian Knowledge Tracing (~1–2 pages). Corbett & Anderson (1995). The four parameters (P(L₀), P(T), P(S), P(G)), the posterior update, the transition to next opportunity. Why it fits sparse lab data better than deep knowledge tracing.
  4. 2.4 Item Response Theory — 1PL / Rasch (~1 page). The 1-parameter logistic model: P(correct) = σ(θᵢ − bⱼ). Why you estimate θ per student and b per question. Why 1PL and not 2PL or 3PL for this dataset size.
  5. 2.5 Cohort normalisation & Bayesian shrinkage (~1 page). James-Stein, empirical Bayes, hierarchical models. The core intuition: with n=2 submissions you shouldn't trust a student's raw mean — shrink it toward the class mean until evidence accumulates.
  6. 2.6 Collaborative filtering in education (~½–1 page). User-user kNN, cosine similarity, the "students like you were flagged" pattern. Why CF is a fallback here, not primary.

§2.4 — IRT 1PL (Rasch). Probability that student i answers question j correctly, parameterised by student ability θᵢ and question difficulty bⱼ:
P(correct | θᵢ, bⱼ) = σ(θᵢ − bⱼ) = 1 / (1 + exp(−(θᵢ − bⱼ)))
§2.3 — BKT posterior update after one observation, followed by the transition to the next opportunity:
If correct: P(L|obs) = P(L)·(1 − P(S)) / [ P(L)·(1 − P(S)) + (1 − P(L))·P(G) ]
If wrong: P(L|obs) = P(L)·P(S) / [ P(L)·P(S) + (1 − P(L))·(1 − P(G)) ]
Transition: P(Lnext) = P(L|obs) + (1 − P(L|obs))·P(T)
§2.5 — Bayesian shrinkage (empirical Bayes). The shrunk score S_final pulls a student's raw composite toward the class mean by a factor that decays as the sample size n grows; K = 5:
w_n = n / (n + K)
S_final = w_n · S + (1 − w_n) · class_mean
