\subsection{Implementation Overview} \label{sec:overview}

The dashboard is deployed in two versions, both are fully featured and runnable in parallel from a single working tree. 
Version~1 (\texttt{code/}) is a Streamlit implementation that consists of two separate apps: one for instructors and one for lab assistants, each running their own Python process. 
Version~2 (\texttt{code2/}) is a single app, FastAPI process that serves both an asynchronous HTTP API at \texttt{/api/} and two built React single-page applications (SPAs) for instructors and assistants respectively. 
V2 builds on V1, with every feature from V1 reimplemented in V2, additional features added on top and performance improvements.

An identical analytical design sits behind both iterations of the dashboard. 
Data ingestion, scoring, the advanced model layer, RAG feedback and saved sessions are realised in V1 as the \texttt{learning\_dashboard/} package under \texttt{code/}, and in V2 within the \texttt{backend/} package under \texttt{code2/}. V2 splits V1's monolithic \texttt{analytics.py} into per-signal files and adds a thin infrastructure layer (Pydantic schemas, dependency injection, a disk-backed scoring cache) required by the stateless request model. 
The data pipeline, the runtime-state sharing, and the threshold-classification machinery (the bands and the \texttt{classify\_score} helper) carry across both versions unchanged. 
The scoring outputs, however, have developed in several places: V2 fits a 2PL IRT model that also estimates student abilities where V1 fits a Rasch (1PL) model,
V2 fits BKT parameters once at FastAPI startup with the \texttt{lifespan} handler and holds them in an in-process cache, so the four-parameter model is fit once per process lifetime and reused across stateless requests where V1 leaves the parameters at their configured defaults from \texttt{config.py} (unless changed through the Settings sliders),
V2's improved struggle model adds an IRT-residual difficulty adjusted path (\texttt{expected\_correct} - \texttt{observed\_correct} using the 2PL ability and discrimination) that V1 has no equivalent for, and V2 weights the baseline struggle signal by per-row measurement confidence where V1 takes the unweighted mean. 
Both versions share a single runtime-state file, \texttt{data/lab\_session.json}, coordinated through the \texttt{filelock} library under a five-second exclusive lock, so that a session started in V1 becomes visible to V2 within a few seconds and vice versa. 
The implication is that the analytical contribution is decoupled from the presentation layer, and where V2 extends V1 it does so within the same analytical framework with the same signal categories, the same threshold bands, the same combination structure, whilst refining the numerics that feed into each.

The motivation behind the second iteration of the dashboard is highlighted later on in the section. The first iteration had a Streamlit foundation that was ideal for rapid iteration and early user feedback, but was limiting in terms of performance, scalability and flexibility. 
The limitation of rerunning the entire script on every interaction meant that the system struggled to maintain a responsive user experience which complicates asynchronous work. 
The second iteration addresses these limitations by moving the presentation layer to React and the backend to FastAPI whilst keeping the analytical core untouched. 
The comparison between V1 and V2 is a recurring theme in the implementation section, as it provides a useful lens to explain the rationale behind design decisions and the trade-offs involved. 

This chapter is organised top-down so that Section \ref{sec:techstack} describes the tech stack used by both versions. 
Section \ref{sec:structure} describes the runtime processes, the shared runtime state and the deferred-actions pattern that underpins V1's reactive interactions. 
Section \ref{sec:pipeline} and \ref{sec:session} cover the data pipeline from the lab endpoint to the in-memory dataframe and the live and saved-session lifecycles. 
Section \ref{sec:analytics} covers the baseline analytics and Section \ref{sec:advanced} covers the advanced modelling layer of measurement confidence, item-response-theory difficulty, Bayesian knowledge-tracing mastery, and the improved struggle model that combines them. 
Section \ref{sec:rag} covers the retrieval-augmented feedback layer that produces real-time coaching suggestions for the lab assistant.
Section \ref{sec:views} and \ref{sec:assistant} walk through the instructor views and the lab-assistant flows respectively. 
Section \ref{sec:problems} highlights the implementation problems encountered and the resolutions adopted.


\subsection{Technology Stack} \label{sec:techstack}

The analytical core, written in Python 3.11, is shared between both versions of the dashboard whilst relying on pandas and NumPy for in-memory data manipulation, SciPy for model fitting, ChromaDB and sentence-transformers for the RAG layer, and the OpenAI Python SDK for LLM interactions. 
The presentation and service layers are where the systems diverge, with V1 using Streamlit for the instructor and assistant dashboards and V2 using FastAPI to serve a React single-page application built with Vite and TypeScript.

\subsubsection{V1 - Instructor Process (Streamlit)} 

The V1 instructor process is entered through \texttt{code/app.py}, a thin wrapper that calls \texttt{main()} entry point in \texttt{learning\_dashboard.instructor\_app}. 
Streamlit's execution model is single-threaded meaning that simultaneous users each receive their own server-side \texttt{st.session\_state} keyed by Streamlit's session identifier.
The sidebar drives both the filtering (temporal and module filters) and the live-session controls (start, end, the six-character lab code and the joined-assistants roster). 
View routing is held in the \texttt{current\_view} session-state key, updated through a callback whenever the user picks from the view selector; the main pane renders one of the seven views described in Section \ref{sec:views} based on this value.


\subsubsection{V1 - Assistant Process (Streamlit, mobile)} 

The V1 lab assistant process is entered through \texttt{code/lab\_app.py}, a thin wrapper around the \texttt{main()} entry point in \texttt{learning\_dashboard.assistant\_app}, intended for phone-sized viewports.
Assistant identity is persisted across reloads through a URL \texttt{?aid=<slug>\_<nnnn>} query parameter (for example, \texttt{?aid=alice\_4827}), set when the assistant first joins and read on every subsequent render.
The page auto-refreshes every five seconds via the \texttt{st\_autorefresh} helper from \texttt{streamlit-autorefresh}, independently of the instructor app's refresh cycle.
Each render reads the shared lab-session state described in Section \ref{sec:shared-state} and forwards to one of three views: a join screen when no \texttt{aid} is present (Figure \ref{fig:asst-join}), an unassigned standby view when an \texttt{aid} is present but no student is assigned (Figure \ref{fig:asst-unassigned}) or the assigned view when the assistant has been assigned to a student (Figure \ref{fig:asst-assigned}).

\subsubsection{V2 - React + FastAPI} \label{sec:v2-react-fastapi}

V2 further refines V1's architecture by addressing four limitations encountered after V1 reached feature completion.
Streamlit's execution model reruns the entire script on every interaction, so long-running operations such as the RAG collection build or the BKT Maximum Likelihood Estimation (MLE) fit block the user-interface thread and force the cold start mitigation described later in Section \ref{sec:pipeline} and Section \ref{sec:analytics}.
There is no native support for asynchronous code, so awaiting an OpenAI or ChromaDB response would block the entire app, which is particularly problematic for the lab assistant's real-time coaching suggestions.
The theming model is limited in contrast to the seven-theme palette and accent colour toggle added in V2, and implementing a custom theme switcher in Streamlit would have required a non-trivial amount of custom CSS and JavaScript injection.
Multi-page navigation in Streamlit is route-based with a full-page reload, so V1 works around this by holding the active view in session state and switching the rendered pane on each rerun.


V2 consolidates the two V1 processes into a single FastAPI process entered through \texttt{code2/backend/main.py}. 
The same process exposes the analytical core at \texttt{/api/*} and serves two HTML entry points from the built \texttt{frontend/dist/} directory: \texttt{index.html} is the instructor counterpart to V1's \texttt{code/app.py}, and \texttt{mobile.html} is the lab-assistant counterpart to V1's \texttt{code/lab\_app.py}.
The two entry points share the same React bundle but have different route components and separate routing trees.
View-level descriptions for both flows live in Section \ref{sec:views} and Section \ref{sec:assistant}, with the V1 versus V2 comparison captured in Table \ref{tab:views-comparison}.

Eleven routers under \texttt{code2/backend/routers/} adapt the analytical core to HTTP: \texttt{live}, \texttt{student}, \texttt{question}, \texttt{analysis}, \texttt{sessions}, \texttt{settings}, \texttt{models\_cmp}, \texttt{lab}, \texttt{rag}, \texttt{meta} and \texttt{cf}. 
The \texttt{mobile.html} bundle calls \texttt{/api/lab/*}; the \texttt{index.html} bundle calls the others. 
The frontend is Vite, React and TypeScript with nine top-level views, eight of them are on the instructor side and one for the lab assistant; the eight instructor views include all the ones V1 carries with the addition of a \texttt{SessionProgression} view that replays struggle evolution across the session, an interaction pattern V1's rerun model cannot express.
V2 also adds a seven-theme runtime palette (paper, newsprint, solar, scifi, blueprint, matrix, cyberpunk) swappable from the settings view through React Context and CSS custom properties; the accent colour is picked separately and persisted to \texttt{localStorage}.
A FastAPI \texttt{lifespan} handler runs at startup and prewarms the data frame, incorrectness scores, the struggle and difficulty leaderboards, the BKT and IRT fits and the ChromaDB embedding collection so that the first user request is not subject to cold-start latency. Figure \ref{fig:v2-inclass} shows the V2 In-Class view in the scifi theme.

\subsubsection{Shared Runtime State} \label{sec:shared-state}



All three apps share a single runtime-state file at \texttt{data/lab\_session.json}, accessed through the \texttt{filelock} library under a five-second exclusive lock.   
Writes are atomic, meaning that the new state is written to a \texttt{.tmp} file first and then renamed via \texttt{os.replace}, so a reader never observes a partial JSON file (Figure \ref{fig:code-lablock}).
The file holds the active session metadata, the roster of joined assistants keyed by identifier, and an assignment map from student to assistant with a \texttt{helping} or \texttt{helped} status.
A change made by any process becomes visible to the others within one refresh cycle.

def _read_state_unlocked() -> dict[str, Any]:
    state_file = paths.lab_session_path()
    if not state_file.exists():
        return _default_state()
    try:
        raw = json.loads(state_file.read_text(encoding="utf-8"))
    except (OSError, json.JSONDecodeError):
        return _default_state()
    return _normalize_state(raw)
\end{lstlisting}

\begin{lstlisting}[firstnumber=142]
def _write_state_unlocked(state: dict[str, Any]) -> None:
    state_file = paths.lab_session_path()
    state_file.parent.mkdir(parents=True, exist_ok=True)
    normalised = _normalise_state(state)
    tmp_file = state_file.with_suffix(".tmp")
    tmp_file.write_text(json.dumps(normalised, indent=2), encoding="utf-8")
    tmp_file.replace(state_file)

\subsubsection{Deferred-Actions Pattern}

% Streamlit forbids mutating \texttt{st.session\_state} after a widget has been instantiated in the same run; doing so raises a \texttt{StreamlitAPIException}.
% V1 sidesteps this constraint through a deferred-actions pattern: callbacks and post-widget click handlers do not mutate session state directly, but instead set a \texttt{pending\_*} flag on session state (for example, \texttt{pending\_session\_load\_record}, \texttt{pending\_return\_to\_live\_data} or \texttt{pending\_remove\_assistant\_id}) describing the change to be made.
% The next rerun reads those flags at the top of \texttt{main()}, before any widget is constructed, applies the mutation and clears the flag.
% This avoids extra threads or queues for state coordination, and keeps each transition atomic in the sense that one rerun applies one change.
% V2 does not need this pattern, as React and FastAPI do not impose the same widget-instantiation rule.

Streamlit forbids modifying \texttt{st.session\_state} after a widget has been instantiated in the same run; doing so raises a \texttt{StreamlitAPIException}. 
V1 deals with this constraint through a deferred-actions pattern where callbacks and post-widget click handlers do not modify the session state directly, but instead set a \texttt{pending\_*} flag on the session state describing the change to be made.
The next rerun reads those flags inside \texttt{main()}, before the sidebar widgets that consume these flags are constructed, applies the changes and clears the flag.
This pattern avoids the need for extra threads or queues for state coordination, and keeps each transition atomic in the sense that one rerun applies one change (Figure \ref{fig:code-deferred}).
V2 does not need this pattern, as React and FastAPI do not impose the same widget-instantiation rule.

\subsection{Data Pipeline} \label{sec:pipeline}


The data pipeline is a five-step path from the lab endpoint to the in-memory dataframe which gets consumed by the scoring layer: endpoint polling, detection of the JSON-vs-XML payload, parse, normalise, cache.
Both V1 and V2 use the same modules for the first four steps and diverge only at the caching layer, where V1 relies on Streamlit's \texttt{@st.cache\_data} decorator and an in-process dictionary whilst V2 layers \texttt{cachetools} time-to-live (TTL) caches over a disk-persisted incorrectness cache that survives process restarts (Figure \ref{fig:pipeline-flow}).

\paragraph{Endpoint retrieval and parsing}
The endpoint is Loughborough's existing PHP retrieval URL, named in \texttt{config.py} as \texttt{API\_URL}; the same URL is used by both versions.
The HTTP fetch is a synchronous \texttt{requests.get()} call with a 30-second timeout (\texttt{API\_TIMEOUT = 30}).
Under normal operation the endpoint returns newline-delimited JSON, with a pure-XML payload retained as a fallback for legacy endpoint behaviour; format detection is a single-attempt JSON parse on the first non-empty line.
Each JSON record carries the outer fields \texttt{module}, \texttt{question}, \texttt{session} and \texttt{user}, plus an inner \texttt{xml} string.
The inner XML carries one or more \texttt{<submission>} elements, each with a \texttt{timestamp} attribute, an \texttt{<srep>} element (the student's answer), and a \texttt{<feedback>} element (the AI feedback text consumed downstream by the incorrectness scorer); the pipeline emits one DataFrame row per submission element.
Embedded XML is parsed with the standard-library \texttt{xml.etree.ElementTree} (Figure \ref{fig:code-parsexml}).

\paragraph{Data normalisation and structuring}
All records collapse into a single pandas \texttt{DataFrame} with seven columns: \texttt{module}, \texttt{question}, \texttt{timestamp}, \texttt{student\_answer}, \texttt{ai\_feedback}, \texttt{session} and \texttt{user}.
An eighth column, \texttt{academic\_period}, is added by \texttt{academic\_calendar.add\_academic\_period\_column()} and maps each row's timestamp to a Loughborough 2025/26 academic-week label which is consumed by the Data Analysis view in Section \ref{sec:views}.
The excluded modules filter (\texttt{EXCLUDED\_MODULES}) drops test and admin modules before any scoring and modelling steps.
Timestamps are parsed with \texttt{pandas.to\_datetime(\dots, errors="coerce")}, rows with unparseable timestamps are dropped and the DataFrame is sorted by timestamp.
% A boolean \texttt{has\_feedback} column is then added by \texttt{add\_feedback\_flag()} and consumed by the incorrectness scorer in Section \ref{sec:analytics}: rows with empty \texttt{ai\_feedback} are short-circuited to the neutral fallback $x_i = 0.5$ rather than dispatched to the OpenAI batch call.

\paragraph{Caching}
V1 caches at two layers.
\texttt{fetch\_raw\_data()} is wrapped in \texttt{@st.cache\_data(ttl=CACHE\_TTL)} with \texttt{CACHE\_TTL = 10} seconds, so successive sidebar reruns within the same refresh cycle do not re-hit the endpoint.
A session-state shortcut in \texttt{load\_data()} hashes the raw response and skips the parse and normalise steps when the hash is unchanged from the previous render.
The incorrectness scorer in Section \ref{sec:analytics} maintains its own in-process dictionary \texttt{\_incorrectness\_cache} in \texttt{analytics.py}, which is keyed by feedback text; it is local to the process and does not survive a restart.
The Streamlit auto-refresh widget runs on a separate cadence (\texttt{AUTO\_REFRESH\_INTERVAL\_DEFAULT = 300} seconds, configurable from five seconds up to five minutes), the 10-second \texttt{CACHE\_TTL} deduplicates within one refresh cycle, whilst the chosen auto-refresh interval controls how often a fresh cycle happens.

V2 retains the same in-process scorer cache but enhances the storage in three places.
The raw-DataFrame cache becomes a \texttt{cachetools.TTLCache} with \texttt{maxsize=1} and a 60-second TTL (\texttt{CACHE\_TTL = 60} in \texttt{code2/backend/config.py}), since FastAPI's stateless request model would otherwise rebuild the DataFrame on every endpoint hit.
Above that, four additional \texttt{TTLCache} layers sit one level higher in the call stack: \texttt{\_struggle\_cache} and \texttt{\_difficulty\_cache} at 15 minutes, \texttt{\_improved\_cache} and \texttt{\_irt\_difficulty\_cache} at 10 minutes, each caching the scoring outputs per session identifier.
On top of both, V2 also adds a disk-persisted incorrectness cache at \texttt{data/incorrectness\_cache.json}, written through \texttt{code2/backend/incorrectness.py}.
A FastAPI \texttt{lifespan} handler runs at startup, loads the DataFrame, computes the warm ratio (the fraction of pending feedback strings already on disk) and if the warm ratio is at least 0.95 it short-circuits the scoring pass entirely, otherwise it batches the missing feedback strings to OpenAI and updates the cache file (Figure \ref{fig:code-warmratio}).
The same handler then fits the BKT and IRT models once and stashes their parameters in memory for reuse across requests.

\subsection{Session Management} \label{sec:session}

A session in this system is a bounded window of submissions used for the in-class workflow, which is recorded so it can be revisited later on.
The live session lives in \texttt{data/lab\_session.json} (covered in Section \ref{sec:shared-state}); once ended, the record is appended to \texttt{data/saved\_sessions.json} for review later in the Previous Sessions view.

The session lifecycle is coordinated through the file-locked state from Section \ref{sec:shared-state}, with every mutation atomic under the write lock.
\begin{enumerate}
    \item The instructor clicks ``Start Session'' in the sidebar, which calls \texttt{lab\_state.start\_lab\_session()}; this generates a six-character code with \texttt{generate\_session\_code()} and writes the session metadata (\texttt{session\_code}, \texttt{session\_active=True}, \texttt{session\_start}, \texttt{class\_id}, \texttt{class\_label}) under the file-locked write helper.
    \item Assistants enter the code and a display name on the lab assistant app, which calls \texttt{join\_session(session\_code, name)} and either mints a new \texttt{assistant\_id} or reuses an existing one if a matching display name is already in the roster, then appends them to the roster.
    \item Assignment is bidirectional: the dashboard calls \texttt{assign\_student(student\_id, assistant\_id)} or, if the lab instructor has enabled self-allocation, the assistant calls \texttt{self\_claim\_student(student\_id, assistant\_id)} from the lab app; both adjust the same assignment map under the write lock, and completion is signalled by \texttt{mark\_student\_helped(student\_id)}, which stamps \texttt{status="helped"} and \texttt{helped\_at}.
    \item Ending the session calls \texttt{end\_lab\_session()}, which sets \texttt{session\_active=False}, clears the \texttt{assignments} map and nulls each assistant's \texttt{assigned\_student}; the session is then optionally persisted as a saved record.
\end{enumerate}
The session code is drawn uniformly from \texttt{string.ascii\_uppercase + string.digits}, giving $\log_2(36^6) \approx 31$ bits of entropy, which is more than sufficient for the expected scale of use and allows for a short and human-friendly code.
During a live session the instructor sidebar shows the join code on a card with the elapsed time in \texttt{HH:MM:SS} format, the class label and an ``End Session'' button (Figure \ref{fig:session-live}).

    \caption{Instructor sidebar during a live session, showing the join code, the elapsed-time timer and the End Session button.}

Each ended session is appended to \texttt{data/saved\_sessions.json} through \texttt{save\_session\_record(record)}; each record carries an \texttt{id} (uuid), a display \texttt{name}, \texttt{start\_time}, \texttt{end\_time}, \texttt{duration\_seconds}, \texttt{class\_id}, \texttt{class\_label} and a nested \texttt{context} dict (Figure \ref{fig:saved-session-json}).
The \texttt{context} dict carries only the UI filter state; the assistant roster and assignment map are not persisted, since they live in the live-state file \texttt{data/lab\_session.json} which is cleared on \texttt{end\_lab\_session()}.
This is a deliberate scope split: saved sessions are for retrospective analysis of submissions in a time window and not for replaying assistant allocations.
CRUD operations live in \texttt{data\_loader.py} in both versions and persist through \texttt{\_write\_saved\_sessions\_payload()}, which uses the same atomic \texttt{.tmp} $\to$ \texttt{path.replace(\dots)} pattern as the live-state writer (Figure \ref{fig:code-lablock}). 
In V1, \texttt{apply\_saved\_session\_to\_state(record, available\_modules)} restores the saved filter state to \texttt{st.session\_state} through the deferred-actions \texttt{pending\_session\_load\_record} flag covered in Section \ref{sec:structure}, the next rerun reads the flag, applies the filter state and routes back to the In-Class view at that time window. V2 has no equivalent server-side function; the React Previous Sessions view re-applies the filter state client-side from the \texttt{GET /api/s/\{id\}} response, sidestepping Streamlit's widget-instantiation rule entirely.

\subsection{Analytics Implementation} \label{sec:analytics}

The analytical core runs five scoring pipelines on each cache refresh: incorrectness, student struggle, question difficulty, collaborative filtering (CF) and mistake clustering.
Incorrectness is computed first because struggle, difficulty and CF all consume the resulting $x_i \in [0,1]$ score. 
V1 keeps these five pipelines in a single \texttt{analytics.py} module, whereas V2 splits them across per-signal modules (\texttt{incorrectness.py}, \texttt{struggle.py}, \texttt{difficulty.py}, \texttt{collab.py}, \texttt{clustering.py}) while keeping the public function signatures and the threshold-classification machinery unchanged; the per-signal refinements V2 introduces are flagged inline in the subsections below.

\subsubsection{Incorrectness Scoring}

\textbf{Maths recap.}
Each submission $i$ receives a scalar incorrectness $x_i \in [0, 1]$ where $x_i = 0$ means completely correct and $x_i = 1$ means completely incorrect.
The mapping from the \texttt{ai\_feedback} text to $x_i$ uses the gpt-4o-mini model in the OpenAI API. 
Empty feedback returns the neutral score of $x_i = 0.5$ without an API call. 

Incorrectness is computed by \texttt{compute\_incorrectness\_column(df)}, which routes uncached feedback strings through \texttt{\_call\_openai\_batch()} in groups of up to \texttt{OPENAI\_BATCH\_SIZE = 20} feedback texts per request - smaller batches parse more reliably than the API's nominal maximum.
A cold-start guard \texttt{SCORING\_PER\_RUN\_CAP = 500} caps the new pairs scored per Streamlit rerun in V1 or per FastAPI cache rebuild in V2; any feedbacks beyond the cap fall back to the neutral $0.5$ and are scored on subsequent runs, keeping the UI responsive when the cache is empty.
On API failure the row falls back to $0.5$ as well and the bad response is not persisted; the next rerun retries the call (Figure \ref{fig:code-openai}).

Both versions keep the in-process dictionary \texttt{\_incorrectness\_cache} keyed by feedback text, introduced in Section \ref{sec:pipeline}.
V2 takes this a step further with a disk-persisted JSON at \texttt{data/incorrectness\_cache.json} whose payload is model-versioned: the \texttt{OPENAI\_MODEL} name is stored alongside the scores and the cache is discarded on load when the model name changes, protecting scores from being reused when the model is changed.
V2's \texttt{compute\_incorrectness\_column} also exposes two kwargs used by the lifespan handler: \texttt{max\_new\_scores} overrides the per-run cap (the prewarm sets it to lift the cap entirely on a cold boot), and \texttt{score\_new=False} is a lookup-only mode that skips OpenAI calls and returns $0.5$ for any missing feedback. The warm-ratio short-circuit shown in Figure \ref{fig:code-warmratio} is the lifespan handler exercising \texttt{score\_new=False}.

\subsubsection{Baseline Student Struggle Model}

\textbf{Maths recap.}
Each student $s$ scores
\[
S(s)
= \alpha\, N_s
+ \beta\, T_s
+ \gamma\, I_s
+ \delta\, R_s
+ \eta\, A_s
+ \zeta\, D_s
+ \theta\, \mathrm{Rep}_s,
\]
where each capital symbol is the min-max normalisation of its raw per-student signal across the cohort; the mapping back to the Chapter \ref{sec:design} hat-tilde notation is given in Table \ref{tab:struggle-7sig}.
A Bayesian shrinkage step then pulls students with few submissions toward the cohort mean (full derivation in Section \ref{sec:design} and Appendix E):
\[
S_{\mathrm{final}}(s) = w_n\, S(s) + (1 - w_n)\,\bar{S}, \qquad w_n = \frac{n}{n + K}, \qquad K = 5,
\]
where $\bar{S}$ is the cohort mean of $S$.
The recent-incorrectness signal $A$ uses an exponential time decay $w_i = e^{-\lambda \Delta t}$ with half-life $T_{1/2} = 1800$ s, giving $\lambda = \ln 2 / 1800$.

The seven signals and their default weights are listed in Table \ref{tab:struggle-7sig}; an import-time assertion in \texttt{config.py} enforces the weight sum to $1.0$.
Each per-student aggregate is computed once over a single \texttt{groupby("user")} pass; the seven signals are then min-max normalised across the cohort and combined under the weighted sum above as vector operations on the DataFrame.
Bayesian shrinkage is the final step: a student with $n \gg K$ submissions is unaffected, whereas a student with $n \ll K$ is pulled toward the cohort mean, which stops a single noisy submission from putting a low-volume student at the top of the leaderboard (Figure \ref{fig:code-shrinkage}).

V1 takes the unweighted arithmetic mean when computing both the overall incorrectness signal $I$ and the recent-incorrectness signal $A$.
V2 weights both by the per-row measurement confidence introduced in Section \ref{sec:advanced}, so a submission with low-confidence feedback contributes less to the incorrectness signals and therefore to the struggle score; this is an improvement since the OpenAI scorer can be noisy on short or ambiguous feedback texts.
Confidence-weighted aggregation is one of several analytical refinements introduced in V2 (see Section \ref{sec:advanced} for the IRT and BKT additions).
Threshold bands live in \texttt{STRUGGLE\_THRESHOLDS}: On Track ($0.00$--$0.20$), Minor Issues ($0.20$--$0.35$), Struggling ($0.35$--$0.50$) and Needs Help ($0.50$--$1.00$); the colours match the In-Class leaderboard colours shown in Section \ref{sec:views}.

\subsubsection{Baseline Question Difficulty Model}

\textbf{Maths recap.}
Each question $q$ scores
\[
D(q) = \alpha\, C_q + \beta\, T_q + \gamma\, A_q + \delta\, F_q + \epsilon\, P_q,
\]
where each capital symbol is the min-max normalisation of its raw per-question signal across the question set; the mapping back to the Chapter \ref{sec:design} tilde notation is given in Table \ref{tab:difficulty-5sig} (full derivation in Section \ref{sec:design} and Appendix E). The Greek weights are reused across the struggle and difficulty models in line with Section \ref{sec:design}; the formulas are otherwise independent.

The five signals and their weights are listed in Table \ref{tab:difficulty-5sig}; an import-time assertion in \texttt{config.py} enforces the weight sum to $1.0$.
V1 normalises each signal globally across the full question set using \texttt{min\_max\_normalise}; V2 instead normalises within each module using \texttt{min\_max\_normalise\_grouped} keyed by \texttt{module}.
A hard question in one module may not be the same as a hard question in another, and a global normalisation would let one module's difficulty distribution dominate the others - V2's grouped pass restores per-module sensitivity without changing the weights themselves (Figure \ref{fig:code-permodule}).

\subsubsection{Collaborative Filtering}

\textbf{Maths recap.}
For each pair of students $(s, t)$, similarity is the cosine over the five-element behavioural feature vector $\mathbf{v}_s = (N_s, T_s, I_s, A_s, D_s) \in [0,1]^5$ (using the cleaned-up symbols from Table \ref{tab:struggle-7sig}):
\[
\mathrm{sim}(s, t) = \frac{\mathbf{v}_s^{\top} \mathbf{v}_t}{\lVert \mathbf{v}_s \rVert_2 \, \lVert \mathbf{v}_t \rVert_2}.
\]
A struggling student $s^\star$ flagged by the parametric model "infects" any peer $t$ with $\mathrm{sim}(s^\star, t) \geq \tau$ and $t$ not already flagged, surfacing students whose behaviour looks like a struggler's even when their parametric score does not yet cross the threshold.

The five-element feature vector \texttt{CF\_FEATURES} carries the four normalised struggle signals $N$, $T$, $I$, $A$ and the improvement-slope $D$ (named \texttt{n\_hat}, \texttt{t\_hat}, \texttt{i\_norm}, \texttt{A\_norm}, \texttt{d\_hat} respectively in code).
The algorithm considers $k = 3$ nearest neighbours per flagged struggler.
The similarity threshold $\tau$ defaults differ between versions, with V1 at $\tau = 0.6$ and V2 at $\tau = 0.7$; the higher default trades recall for tighter precision, and a slider in the Settings view (Section \ref{sec:views}) allows the user to toggle $\tau$.
The output is a list of (similar student, source student, similarity score) tuples rendered in the In-Class view's CF diagnostic panel beneath the leaderboards if the collaborative-filtering flag \texttt{cf\_enabled} is set to \texttt{True}.



\subsubsection{Mistake Clustering}

For each question with at least \texttt{CLUSTER\_MIN\_WRONG = 3} incorrect submissions, \texttt{cluster\_question\_mistakes()} TF-IDF vectorises the wrong-answer texts and clusters them with KMeans.
The number of clusters $k$ is selected by silhouette score, capped at \texttt{CLUSTER\_MAX\_K = 5}, after which each cluster is labelled by a short OpenAI call (\texttt{\_label\_clusters\_with\_openai}) that turns the centroid and a few representative answers into a human-readable tag.
Up to \texttt{CLUSTER\_MAX\_EXAMPLES = 3} representative answers are shown per cluster, each truncated to \texttt{CLUSTER\_EXAMPLE\_MAX\_CHARS = 300} characters for the UI.
The cache key uses SHA1 of the input texts rather than Python's built-in \texttt{hash()}; the built-in is salted per process and would invalidate the cache on every restart, defeating the cache's purpose.
Results surface in the Question Detail view (Section \ref{sec:views}) as labelled clusters next to the question text.


\subsection{Advanced Model Implementation} \label{sec:advanced}

The four models in this section sit behind the Settings view's Improved Models toggle, which switches the In-Class leaderboard between the baseline and improved struggle models, and the Question Detail difficulty between the baseline and IRT versions.
All four share an implementation pattern: building a response matrix or per-row feature from the live DataFrame, fitting by maximum likelihood where applicable, and falling back to baseline results when the fit declines so the UI remains responsive in edge cases.
V1 wires the models statically, whilst V2 fits at FastAPI startup with the \texttt{lifespan} handler and caches the fitted parameters per process.
The improved struggle model is the one place where the upstream signals are combined into a single score.
\subsubsection{Measurement Confidence}
\textbf{Maths recap.}
For an incorrectness score $x \in [0, 1]$ with feedback text of length $\ell$ characters, confidence is
\[
\kappa = \mathrm{ramp}_{\ell, 0, L} \cdot \big(0.5 + 0.5\,|2x - 1|\big) \cdot \kappa_{\text{base}},
\]
where $\mathrm{ramp}_{\ell, 0, L}$ rises linearly from $0$ to $1$ over $[0, L]$ characters ($L = 20$ from \texttt{MEASUREMENT\_CONFIDENCE\_MIN\_LENGTH}), $|2x - 1|$ is the extremity term (1 at the ends of $[0,1]$, 0 at $x = 0.5$), and $\kappa_{\text{base}} = 0.7$ (\texttt{MEASUREMENT\_CONFIDENCE\_BASE}).

Confidence is a meta-signal that quantifies how much the system trusts a given scorer verdict; short or empty feedback texts and verdicts near the middle of the scale are flagged as less trustworthy than long, decisive ones.
Only two factors feed the formula: feedback length and extremity, combined under the base ceiling shown in Figure \ref{fig:code-confidence}.
Empty feedback returns $\kappa = 0$ rather than the neutral $0.5$ used for the underlying incorrectness score; an empty score is fallback, not measurement.
V2 introduces \texttt{\_CONFIDENCE\_FLOOR = 0.2} inside \texttt{\_confidence\_weighted\_mean()} so that low-confidence rows still contribute proportionally to the aggregate; V1 has no equivalent floor.

\subsubsection{Item Response Theory (IRT) Difficulty}

\textbf{Maths recap.}
V1 fits a one-parameter logistic (Rasch) model
\[
P(X_{s,q} = 1 \mid \theta_s, \beta_q) = \sigma(\theta_s - \beta_q),
\]
where $\theta_s$ is the student ability, $\beta_q$ the question difficulty and $\sigma$ the logistic sigmoid.
V2 evolves this to the two-parameter logistic (2PL) form
\[
P(X_{s,q} = 1 \mid \theta_s, \alpha_q, \beta_q) = \sigma\!\big(\alpha_q\,(\theta_s - \beta_q)\big),
\]
adding a positive per-question discrimination $\alpha_q$.
Both fits maximise the Bernoulli log-likelihood across all observed responses via L-BFGS-B \cite{byrdLimitedMemoryAlgorithm1995}.
The 2PL collapses to Rasch when all $\alpha_q = 1$, so V2 is a strict generalisation of V1. 

V1's \texttt{fit\_rasch\_model} builds a response matrix with one row per student and one column per question; the entry is $1$ if the student's best-attempt incorrectness is below the correctness threshold $\tau = 0.5$, $0$ otherwise, and \texttt{NaN} if no attempt exists.
Filtering is iterative: \texttt{IRT\_MIN\_ATTEMPTS\_PER\_QUESTION = 2} drops thin questions and \texttt{IRT\_MIN\_ATTEMPTS\_PER\_STUDENT = 2} drops thin students; the filter is repeated until both minima are satisfied.
The optimiser runs at \texttt{IRT\_MAX\_ITER = 100} iterations and the convergence flag (\texttt{bool(result.success)}) is returned alongside the fit so downstream consumers can detect a failed run.
The fitted $\hat{\beta}_q$ is on the logit scale; the leaderboard maps it to $[0, 1]$ via $D^{\mathrm{IRT}}(q) = \sigma(\hat{\beta}_q)$ and applies the same threshold bands as the baseline difficulty model (\texttt{IRT\_DIFFICULTY\_THRESHOLDS}).

V2's \texttt{fit\_2pl\_model} carries a parameter vector laid out as $[\theta, b, \log a]$ (Figure \ref{fig:code-2pl-unpack}); the discrimination is fitted in log space and exponentiated back, which keeps $\alpha_q > 0$ without an explicit box constraint, and the only box bound applied is $\log a \in [-5, +5]$ to prevent the optimiser parking on degenerate-discrimination items.
The identifiability projection $\bar{\theta} = 0$ and $\overline{\log a} = 0$ (geometric mean of $a$ equal to $1$) is reapplied at every gradient evaluation through the \texttt{\_unpack} step, so the absolute level of the latent parameters is fixed without trading the loss against an arbitrary offset.
The model returns a difficulty DataFrame with \texttt{[question, irt\_difficulty, irt\_discrimination, irt\_difficulty\_level, irt\_difficulty\_color]} and an ability DataFrame with \texttt{[user, theta]}, alongside the convergence flag and the log-likelihood; the whole result is cached at the FastAPI level in \texttt{\_irt\_difficulty\_cache} (\texttt{TTLCache(maxsize=4, ttl=600)}) keyed by the active time window.

The 1PL$\rightarrow$2PL evolution is one of the two algorithmic improvements in V2 over V1, and it carries a free bad-item diagnostic: a fitted $\alpha_q$ near zero flags items that fail to separate the cohort, exposed through the \texttt{irt\_discrimination} column on the Question Detail page.

\subsubsection{Bayesian Knowledge Tracing (BKT) Mastery}

\textbf{Maths recap.}
Bayesian Knowledge Tracing \cite{corbettKnowledgeTracingModeling1995} models each (student, skill) pair as a two-state hidden Markov model.
The hidden state $L_t \in \{\text{not learned},\, \text{learned}\}$ is updated after each observed correctness $c_t \in \{0, 1\}$ by the forward recursion
\[
P(L_t \mid c_t) = \frac{P(L_t)\, P(c_t \mid L_t)}{P(c_t)}, \qquad
P(L_{t+1}) = P(L_t \mid c_t) + \big(1 - P(L_t \mid c_t)\big)\, P(T),
\]
parameterised by the four scalars listed in Table \ref{tab:bkt-defaults}.

V1 leaves the four parameters at their configured defaults whilst V2 fits them once per process at FastAPI startup (the \texttt{lifespan} handler), caching the result in the module-level \texttt{\_BKT\_PARAMS\_CACHE} dictionary so the four-parameter model is fit once per process lifetime and reused across stateless requests.
The fit refuses when fewer than \texttt{BKT\_FIT\_MIN\_OBSERVATIONS = 50} attempts are available, or when every attempt is graded the same way; the optimiser is L-BFGS-B with bounds $[0, 1] \times [0, 1] \times [0, 0.5] \times [0, 0.5]$ where the $0.5$ ceilings on guess and slip implicitly enforce the identifiability constraint $P(G) + P(S) < 1$.

V2's \texttt{fit\_all\_skills} fits one parameter set per skill (module), not per question.
This is a deliberate choice: a question-level fit would require more data than most individual questions ever see, whereas a module-level fit pools across the questions in a single subject area and remains stable on session-sized data.
Predictive AUC via \texttt{sklearn.metrics.roc\_auc\_score} is computed alongside the fit and reported back to the UI.
The mastery threshold \texttt{BKT\_MASTERY\_THRESHOLD = 0.95} is the bar for the ``mastered'' badge on the Student Detail page.

def bkt_update(
    p_mastery: float,
    correct: bool,
    p_guess: float = config.BKT_P_GUESS,
    p_slip: float = config.BKT_P_SLIP,
    p_learn: float = config.BKT_P_LEARN,
) -> float:
    """Single BKT HMM update step.  Returns P(L_{t+1})."""
    if correct:
        p_obs = p_mastery * (1.0 - p_slip) + (1.0 - p_mastery) * p_guess
        if p_obs < 1e-12:
            p_obs = 1e-12
        p_posterior = p_mastery * (1.0 - p_slip) / p_obs
    else:
        p_obs = p_mastery * p_slip + (1.0 - p_mastery) * (1.0 - p_guess)
        if p_obs < 1e-12:
            p_obs = 1e-12
        p_posterior = p_mastery * p_slip / p_obs

    p_next = p_posterior + (1.0 - p_posterior) * p_learn
    return float(np.clip(p_next, 0.0, 1.0))

\subsubsection{Improved Struggle Model}

\textbf{Maths recap.}
The improved struggle score combines three components under a convex sum:
\[
S_{\mathrm{improved}}(s) = w_B\, B_s + w_M\, M_s + w_D\, D_s,
\]
where $B_s$ is the behavioural model (the equal-weighted mean of four cohort-normalised baseline signals from Section \ref{sec:analytics}: $\widehat{A^{\mathrm{raw}}}_s$, $\hat{\tilde{r}}_s$, $\hat{d}_s$ and $\widehat{\tilde{\mathrm{rep}}}_s$), $M_s = \max\!\big(\bar{m}_s - (1 - A^{\mathrm{raw}}_s),\, 0\big)$ is the mastery-gap term derived from BKT (Section \ref{sec:advanced}) - the positive-part difference between mean BKT mastery and the student's recent correctness rate, so the signal only fires when a student underperforms their own mastery estimate - and $D_s$ is a difficulty-adjusted exposure term that reweights each submission by the IRT difficulty of its question.
Default weights are $(w_B, w_M, w_D) = (0.45, 0.30, 0.25)$ from \texttt{IMPROVED\_STRUGGLE\_WEIGHT\_BEHAVIORAL}, \texttt{IMPROVED\_STRUGGLE\_WEIGHT\_MASTERY\_GAP} and \texttt{IMPROVED\_STRUGGLE\_WEIGHT\_DIFFICULTY\_ADJ} respectively (Chapter \ref{sec:design} writes these as $w_{\mathrm{beh}}, w_{\mathrm{mg}}, w_{\mathrm{da}}$).

The three model scores covered in this section (seven-signal baseline struggle, five-signal baseline difficulty, three-weight improved) each carry an optional v2 weight vector loaded at request time from \texttt{data/eval/optimised\_*\_v2.json}, with a fourth JSON (\texttt{optimised\_hyperparams\_v2.json}) carrying the Optuna-tuned values of the two scalar hyperparameters introduced in Chapter~\ref{sec:design}.
Five fields on \texttt{RuntimeConfig} gate the selection: four \texttt{*\_version} string flags (\texttt{struggle\_weights\_version}, \texttt{difficulty\_weights\_version}, \texttt{improved\_struggle\_weights\_version}, \texttt{hyperparams\_version}) and one direct-override scalar (\texttt{shrinkage\_k}) each defaulting to its v1 value.
When \texttt{hyperparams\_version} is set to \texttt{"v2"}, both the shrinkage constant $K$ and the CF threshold $\tau$ are read from the JSON; \texttt{shrinkage\_k} can also accept an integer outside of the v1/v2 values.
A missing or corrupt JSON falls back to v1 with a warning logged.
The toggles are only implemented on the V2 iteration of the dashboard.


V1's \texttt{compute\_improved\_struggle\_scores} builds the three component scores and combines them under the weighted sum above.
The difficulty-adjusted term uses \texttt{incorrectness $\times$ (1 - norm\_diff)} for each submission, so wrong attempts on easy items contribute more than wrong attempts on hard ones - failing an easy question signals struggle more strongly than failing a hard one, since the cohort has already validated the hard item's difficulty.
The distinctive piece is the graceful weight redistribution: if BKT mastery is unavailable for a student, $w_M$ is absorbed entirely into $w_B$ ($w_D$ unchanged); if IRT difficulty is unavailable, $w_D$ is absorbed entirely into $w_B$.
The score is therefore always defined and collapses cleanly to a lower-information variant when one of the component models fails to fit.
Table \ref{tab:improved-redist} lists the four redistribution cases.

V2's \texttt{compute\_improved\_struggle\_scores} takes an additional \texttt{irt\_ability} kwarg.
When the full 2PL parameters are available, \texttt{\_compute\_irt\_residual} replaces the legacy difficulty-adjustment term with the IRT residual shown in Figure \ref{fig:code-irt-residual}.
The intuition is that a student whose observed performance falls below the IRT-expected performance is struggling more than a student tracking expectation, regardless of overall difficulty exposure.
Rows missing $\theta_s$, $\alpha_q$ or $\beta_q$ return \texttt{NaN} and are dropped before aggregation, and the aggregator falls back to V1's legacy term when the full 2PL parameters are absent.
The toggle is plumbed differently between versions: V1 wires \texttt{improved\_models\_enabled} as a session-state boolean derived from the Settings dropdowns, whereas V2 carries the choice in \texttt{runtime\_config.RuntimeConfig.struggle\_model} (``baseline''/``improved'') and \texttt{difficulty\_model} (``baseline''/``irt'') enums, persisted through \texttt{POST /api/settings}.

\subsubsection{v2 Model Refinement Pipeline} \label{sec:v2-training}

The v1 model weights in Sections~\ref{sec:analytics} and~\ref{sec:advanced} were tuned by trial and error in the absence of labelled ground-truth data; the design of the offline pipeline that fits them is set out in Section~\ref{sec:v2-pipeline-design}, and the evaluation is highlighted in Section~\ref{sec:eval-v2-methodology}.
This section describes the implementation.
The four JSON files the pipeline produces under \texttt{data/eval/} are loaded at request time by the toggle in Settings.
v2 vectors are read from disk on the first request after a settings change.

The pipeline is structured as five sequential phases, each implemented with Python scripts in \texttt{scripts/} or as a Jupyter notebook under \texttt{notebooks/}.
Phase~1 (\texttt{eval\_fetch.py}) pulls from the Loughborough submission API once and caches all the responses to a Parquet file under \texttt{data/eval/submissions.parquet}, so later phases are completely offline.
Phase~2 (\texttt{eval\_common.py}) stratifies the cached submissions into snapshots across the lab sessions, twelve cutoff times per session, and the four struggle bands.
Phase~3 (\texttt{eval\_label.py}) calls GPT-4o for each snapshot to obtain a four-band struggle rating, and a four-band difficulty rating for the question side.
The bands "On Track", "Minor Issues", "Struggling", "Needs Help" for struggle; "Easy", "Medium", "Hard", "Very Hard" for difficulty are encoded as an integer index in $\{0, 1, 2, 3\}$ for training.

Phase~4 (\texttt{optimise\_v2\_weights.py} and \texttt{optimise\_hyperparams.py}) fits the v2 vectors. The seven-signal struggle model and the three-weight improved model are fitted under ordinary least-squares linear regression against the band index; the outer scheme is \texttt{GroupKFold(n\_splits=5)} with session as the group identifier, so train and test partitions never share students.
Alternatively, the five-signal difficulty model is fitted under leave-one-out cross-validation due to the smaller sample size.
The two scalar parameters Bayesian shrinkage $K$ and collaborative-filtering similarity threshold $\tau$ are tuned in the two Optuna~\cite{akibaOptunaNextgenerationHyperparameter2019} studies each using the Tree-structured Parzen Estimator~\cite{NIPS2011_86e8f7ab}, with session-grouped five-fold cross-validation Spearman~$\rho$ against the four-band labels as the maximisation objective.

Ordinary least-squares regression was chosen so that the linear functional form of the v1 side is kept so that v1 and v2 are directly comparable term by term, and so that the v2 score remains a single continuous scalar that the dashboard's deployed band thresholds can map to a band label in the same way as the v1 score.
The metrics reported alongside the trained weights are Spearman~$\rho$ for rank quality, linear-weighted Cohen's $\kappa$ for band agreement, and mean absolute band-distance error (MAE).

\subsection{RAG Suggested Feedback} \label{sec:rag}

The RAG layer turns a struggling student's submission history into 2-3 tailored feedback points for the lab assistants, which is surfaced as a panel on the assistant's app.
RAG based feedback beats normal prompting here as it anchors the LLM in the student's actual work rather than producing generic advice, which lets the assistant see coaching points referenced to specific student work rather than generic feedback.
Another application of the RAG generated feedback is in the Question Detail view (Section \ref{sec:views}) where the instructor is pointed towards common failures for the question rather than one student, which lets the instructor give the class more targeted feedback when going through a question rather than a generic walkthrough of it.

\subsubsection{Hybrid Two-Layer Architecture}

\textbf{Maths recap (optional).}
Layer 2 retrieval is a top-$k$ nearest-neighbour search in the 384-dimensional embedding space of \texttt{all-MiniLM-L6-v2}, using cosine similarity through ChromaDB's HNSW index \cite{lewisRetrievalAugmentedGenerationKnowledgeIntensive2020}. 

The two layers correspond to a structured filter followed by a semantic ranker. 
\textbf{Layer 1} is a pandas pre-filter: \texttt{student\_df = df[df["user"] == student\_id]} narrows the corpus to the requested student's own submission history before any embedding work happens, so retrieved evidence is always directly relevant to the student in question and the embedding index is never larger than a single student's history, keeping retrieval latency low. 
\textbf{Layer 2} is a ChromaDB semantic search over the embeddings of those submissions, with a metadata \texttt{where} clause that scopes the search within the active session's collection (\texttt{session\_\{session\_id\}}).
The two-layer output is a small list of (submission text, feedback, score) triples that is then handed to the generator. 
V2's \texttt{code2/backend/routers/rag.py} wraps the synchronous build and generate path in \texttt{asyncio.to\_thread} so the first-time ChromaDB build doesn't block the FastAPI event loop; V1 does not need this as with Streamlit the build happens during the cached scoring pass.
Both the assistant-panel path and the instructor-view path (\texttt{generate\_question\_suggestions}) use the same two-layer retrieval differing only in the Layer 1 pandas filter.

\subsubsection{Embedding Model (all-MiniLM-L6-v2)}

Embeddings are produced locally by \texttt{sentence-transformers} running \texttt{all-MiniLM-L6-v2} (\texttt{RAG\_EMBEDDING\_MODEL}).
This was chosen as the model runs locally with no API call and no per-query cost, and is sufficient for the short submission texts that the lab endpoint produces.
ChromaDB collections are named \texttt{session\_\{session\_id\}} so that embeddings do not leak across sessions and the vector store persists to \texttt{data/rag\_chroma/} with \texttt{chromadb.PersistentClient}.
A short circuit guard \texttt{RAG\_MIN\_SUBMISSIONS = 2} skips the panel for students with fewer than 2 submissions, as the feedback is unlikely to be useful and the embedding work isn't worth doing.
The whole RAG path is gated by \texttt{RAG\_ENABLED\_DEFAULT = True}.

\subsubsection{Generation and Prompting}

The chosen LLM to generate feedback is gpt-4o-mini, which is the same model used for incorrectness scoring, this is due to it being a lightweight, highly optimised model which balances performance and cost. 
The prompt is a system message and a user message that carries the retrieved evidence which includes the student's struggle score, struggle band label, the top three questions ranked by incorrectness and the retrieved question-answer snippets.
Output is constrained to JSON format with OpenAI's \texttt{response\_format=\{"type": "json\_object"\}}, and the prompt itself specifically asks for a JSON array of 2--3 short bullet strings of at most 15 words each.
The parser is permissive in case the model returns a wrapped object instead of a bare array.
Each bullet is addressed to the lab assistant and frames a specific thing to look at.

\subsubsection{Suggestion Caching}

Both versions cache the OpenAI response so that a re-render of the assistant panel doesn't re-call the API, but the caches diverge in both key structure and lifetime.
V1 uses a plain process-local dictionary \texttt{\_suggestion\_cache} keyed by \texttt{student\_id} alone, cleared via \texttt{clear\_suggestion\_cache()} whenever the session changes.
V2 uses a \texttt{TTLCache(maxsize=256, ttl=600)} keyed by the tuple \texttt{(student\_id, window.from\_, window.to\_, window.module, session\_id)}, ageing entries out after ten minutes.
Neither cache invalidates on a submission-count threshold.
This is a deliberate tradeoff as the RAG generation is the most expensive step in the panel and rerunning it on every poll cycle would dominate the OpenAI API cost without a clear improvement in coaching quality.


\subsubsection{Graceful Degradation and Privacy}

The RAG path is wrapped in an optional-dependency guard: a \texttt{\_lazy\_import()} helper attempts to import \texttt{chromadb} and \texttt{sentence\_transformers} and if either is missing the panel doesn't render whilst leaving the rest of the application intact.
\texttt{RAG\_ENABLED\_DEFAULT = True} is followed, but the dependency check has the final say on whether the feedback is generated.
If the OpenAI call fails or times out (V2 sets \texttt{timeout=15.0}; V1 relies on the SDK default), the exception is caught and an empty list is returned; cached bullet points remain on screen and no error message is shown.
Privacy considerations are baked into the design as all retrieval is local; the only data that leaves the process is what appears in the generation prompt (Figure \ref{fig:code-ragprompt}).
The prompt contains the student's struggle score, struggle band label, the top three questions ranked by incorrectness and the retrieved question-answer snippets.
Although the prompt sends this evidence to the LLM, no student identifier of any kind is included - not even the anonymised student IDs that appear in the leaderboards.
The user ID is only used locally for the pandas filter and as a cache key; it never leaves the process.

\subsubsection{In-Class View}

The In-Class view is the dashboard's live session hub, where instructors spend most of their time during a session and will see first on launch. Both versions present the two side-by-side leaderboards where at the top are the top strugglers ordered by descending struggle level and hardest questions ordered by descending difficulty level. 
Each leaderboard is capped at \texttt{LEADERBOARD\_MAX\_ITEMS = 15}, with bar colours following the threshold bands from Section \ref{sec:design}, where with this theme they were adjusted so that grey is Minor Issues/Medium and light blue is On Track/Easy. 
Under the leaderboards all views render the two score-distribution histograms, with a collaborative filtering diagnostic panel appearing alongside them when the CF toggle is enabled in Settings. 
Clicking a bar drills into the corresponding Student or Question Detail view; Streamlit uses \texttt{st.plotly\_chart(on\_select ="rerun")} with a \texttt{point\_index} lookup whilst V2 routes via React Router to \texttt{/student/:id} or \texttt{/question/:id}. 

V2 carries two forms of the In Class view, the default one is the Basic layout which is the one the instructor sees on the first load due to the \texttt{useUiPrefsStore} preference store initialising the \texttt{inClassViewMode} value to \texttt{'basic'}; the layout is selectable through the Settings page's \emph{In-class layout} toggle. 
The basic layout abstracts a lot from the advanced view so that it is a single colour coded bar per student or question, stretched to fill the row using the cohort min-max scale with a minimum visible fill so the lowest score is still visible. 
When switching to the Advanced view the user is able to see more data immediately which includes submissions, their last registered score, the trend in score for the student side of the in class view and for the question side of the in class view they can see the amount of students attempting the question with the average attempts. 
V1 has no equivalent view toggle and its default view is close to what V2's Basic view offers.

\subsubsection{Student Detail View}

When the In-Class layout is set to Advanced, the Student Detail view gives a per-signal breakdown of a single student's struggle score, with one bar for each of the seven signals defined in \ref{sec:analytics} ($N$ submission count, $T$ time active in the session, $I$ mean incorrectness across submissions, $R$ retry rate, $A$ recent incorrectness with a 30-minute exponential decay, $D$ improvement slope and $\mathrm{Rep}$ answer-repetition rate), alongside an hour-of-day distribution of this student's submissions and a list of at most ten of the hardest questions for the student, where each row drills into the corresponding Question Detail page. Additionally there is RAG suggestions here like that which the lab assistant would get. When the Improved Models toggle is enabled in Settings, the view also adds a BKT mastery panel for the modules the student has touched and a decomposition of the Improved Struggle score into its behavioural, mastery-gap and IRT-difficulty-adjusted components from Section \ref{sec:advanced}.

\subsubsection{Question Detail View}

\subsubsection{Data Analysis View}

The Data Analysis view exposes summaries based off all the data pulled from the endpoint and is meant to be used for offline analysis of data rather than in-session use. 
Data is aggregated by the \texttt{academic\_calendar} module to map raw dates into the Loughborough 2025/26 week labels so that the instructor questions phrased in academic time can be answered more swiftly by Loughborough academics. 
A selectbox allows the users to switch between six different chart modes:  module usage, top questions, user activity, activity timeline, activity by academic week and students by module, with the top-questions and user-activity tables capped at \texttt{DATA\_ANALYSIS\_TOP\_QUESTIONS = 10} and \texttt{DATA\_ANALYSIS\_TOP\_USERS = 20} respectively. 
The activity-by-week chart in V2 is rendered as a week-by-day heatmap rather than V1's bar chart; the underlying \texttt{academic\_calendar} mapping is shared between the two versions so the period semantics are identical.

\subsubsection{Settings View}

Both versions expose the runtime model configuration through a Settings page, where they both have common toggles such as Sound Effects (\texttt{SOUNDS\_ENABLED\_DEFAULT = True}), Auto-Refresh (\texttt{AUTO\_REFRESH\_DEFAULT = True}) with the refresh interval selectable from \texttt{AUTO\_REFRESH\_OPTIONS = [5, 10, 15, 30, 60, 120, 300]} seconds, Collaborative Filtering (\texttt{cf\_enabled}) with the similarity-threshold slider $\tau$, and the two model-choice selectboxes. 
When Improved is selected, four sliders appear for the BKT parameters $P(L_0)$, $P(T)$, $P(G)$ and $P(S)$ with defaults from Section \ref{sec:advanced}, a slider change retriggers the per-student mastery computation on the next refresh. 
The two diverge in a few places where V2 has a theme picker that persists to \texttt{localStorage} and an In-Class layout toggle (Basic/Advanced). V1 also exposes a Temporal Smoothing toggle (\texttt{SMOOTHING\_ENABLED}) where V2 hard-codes the EMA at $\alpha = 0.3$ and surfaces \texttt{smoothing\_alpha} only in a read-only configuration reference at the bottom of the page. 
V2 also has an extra Settings section titled "Optimised Weights (v2)", with four select rows: struggle weights, difficulty weights, improved-struggle model, and hyperparams; selecting v2 on any row invalidates the selected cache and reloads the leaderboard on the next refresh.
The difficulty weights selector is disabled when \texttt{difficulty\_model == "irt"} as the IRT path bypasses the model weights entirely, and the improved model selector is disabled when \texttt{struggle\_model != "improved"}.
The per-component verdict for each v2 trained weight vector is reported in Section~\ref{sec:eval-findings}, and the trained weights replace the v1 defaults at request time once a v2 selection is made.

V1 also carries a small UX flourish that V2 does not: six short sci-fi sound cues fired at session start, session end, student selection and other key events, authored as Web Audio API \texttt{OscillatorNode} and \texttt{GainNode} chains in \texttt{code/learning\_dashboard/sound.py} and injected through a zero-height iframe via \texttt{st.components.v1.html()} as Streamlit cannot natively play audio. The cues are gated by the Sound Effects toggle and, whilst not load-bearing analytically, help the instructor track session state without staring at the screen.

\subsubsection{Session Progression View (V2-only)}

\subsection{Lab Assistant System} \label{sec:assistant}

The lab assistant system is the mobile-optimised component of the dashboard that allows lab assistants to get assigned to students, mark them as helped once they have provided support, as well as receive the RAG-suggested feedback for the student they are helping to make more informed decisions about their guidance to students. 
The assistants are coordinated through the shared session file introduced in Section \ref{sec:session}. 
V1's assistant system is shipped as a separate Streamlit app (\texttt{code/lab\_app.py}, delegating into \texttt{code/learning\_dashboard/assistant\_app.py}), V2 ships it as a React mobile route mounted under \texttt{/mobile} on the same FastAPI process that serves the instructor SPA, with the four state views split across \texttt{code2/frontend/src/mobile/views/}. 
Both versions coordinate through the shared \texttt{data/lab\_session.json} file with reads and writes serialised by filelock, so an instructor running V1 and an assistant running V2, or vice versa, see one another's state changes. 
V2 splits the same dispatch functionality across two views, the dedicated screen at  \texttt{code2/frontend/src/views/LabAssistantView.tsx}, and the main form of dispatching which are dropdowns in each row of the In Class leaderboard with the \texttt{AssignCell} component. 
V1 keeps all allocation controls in the instructor sidebar.

\subsubsection{Session Join}

On the first load, both versions present a single screen join form where the assistant enters their name and the six-character session code (\texttt{LAB\_CODE\_LENGTH = 6}) shown on the instructor dashboard and the client POSTs to the shared lab-state file through \texttt{learning\_dashboard.lab\_state.join\_session} (V1) or \texttt{POST /api/lab/join} (V2), receiving an assistant ID. 
Identity is then persisted so that refreshing the browser does not force a rejoin, with the V1 writing the assistant ID into the URL as the \texttt{?aid=} query parameter (which has the side-benefit that a shared link can auto-rejoin) and V2 writing it to \texttt{localStorage} via the small \texttt{readAid}/\texttt{writeAid} helpers in \texttt{mobile/state/aid.ts} so that the URL stays clean. 
V2 also wraps the first \texttt{POST /api/lab/join} in a silent retry loop so that if the assistant cannot join due to the session just starting, the failed join refetches \texttt{/api/lab/state} and retries silently on its own rather than surfacing the error.

\subsubsection{Waiting and Assignment States}

Between assignments the assistant lands on a waiting view that shows their name and active session as well as a button to leave the session. 
The assistants can also see a ranked list of the students, where only students whose struggle has been flagged as "Struggling" or "Needs Help" are shown in the list ordered by descending struggle score. 
Students who are already being helped by another assistant are also not shown to the assistant to avoid confusion. 
The same list is visible to every joined assistant and to the instructor in real time as the lab-state file where lab information is stored, shared and synchronised. 
When the instructor enables the \texttt{allow\_self\_allocation} flag in their controls, each row carries a help student button where the assistant can choose to help a student, when the flag is off (the default), the list is read only and the assistant has to wait for an allocation. 
Both behaviours have a reason to exist as some instructors may want to have more control over the allocation process and prefer to allocate students themselves through the instructor dashboard, whilst others may prefer to let the assistants self-allocate to students as they see fit.

\subsubsection{Live Assistant Allocation and Mark-Helped}

Streamlit widget-instantiation rule & Mutating \texttt{st.session\_state} after a widget has been instantiated raises \texttt{StreamlitAPIException}, breaking the obvious callbacks & Deferred-actions pattern from Section \ref{sec:structure}: callbacks write \texttt{pending\_*} flags into session state, and the top of \texttt{main()} applies them on the next rerun before any sidebar widget is built. \\
\hline
Cross-process write conflicts on the shared lab-state file & Up to three processes (V1 instructor, V1 assistant, V2 FastAPI) can read and write \texttt{data/lab\_session.json} at the same time; without coordination, conflicting or incomplete writes lose updates or corrupt the file. & Every read and write is wrapped in a \texttt{filelock.FileLock} with a configurable timeout, and writes go to a \texttt{.tmp} file that is then \texttt{os.replace()}'d atomically so a partial write is never observable. \\
\hline
OpenAI scoring cold-start cost & On a fresh session the first run would block for tens of seconds whilst every submission was sent to \texttt{gpt-4o-mini} for an incorrectness score. & Two-layer cache: an in-process dictionary satisfies repeat lookups within one process, and a disk-persisted JSON cache at \texttt{data/incorrectness\_cache.json} survives restarts. The prewarm bypasses the OpenAI call entirely when over 95\% of unique feedback strings are already cached. \\
\hline
V2 FastAPI cold-start latency & V2 fits BKT, fits the 2PL IRT, and builds the ChromaDB embedding index at process startup so every subsequent request hits warm caches; running this expensive work synchronously inside the FastAPI startup hook would freeze the event loop for several minutes and block all incoming requests during prewarm. & A \texttt{lifespan} startup handler runs the OpenAI prewarm, the per-skill BKT fits, the 2PL IRT fit, and the ChromaDB build in the background before the server reports ready, and the fitted parameters are cached for the process lifetime. \\
\hline
BKT identifiability on single-class data & When every observed attempt in the window is graded the same way (all correct or all incorrect), the BKT log-likelihood surface degenerates and L-BFGS-B returns nonsensical parameters. & The fit refuses when fewer than \texttt{BKT\_FIT\_MIN\_OBSERVATIONS = 50} attempts are available, or when every attempt is graded identically; the per-skill fit retains the configured defaults and logs a diagnostic explaining the cause. \\
\hline
Per-module difficulty normalisation & V1 min-max normalised the five difficulty signals globally across the entire question set, so one module with a wider raw-signal range could dominate the leaderboard regardless of within-module difficulty. & V2's \texttt{compute\_question\_difficulty\_scores} routes each raw signal through \texttt{min\_max\_normalise\_grouped} keyed by module, restoring per-module sensitivity without changing the weight vector. \\
\hline
Improved-struggle dependence on optional model outputs & The improved struggle score combines behavioural + BKT mastery-gap + IRT-adjusted components under a fixed convex weighting; when either upstream model fails to fit, the score becomes undefined or skewed. & Graceful weight redistribution: if BKT mastery is unavailable, $w_M$ is absorbed entirely into $w_B$; if IRT difficulty is unavailable, $w_D$ is absorbed entirely into $w_B$ (see Table \ref{tab:improved-redist}). The score is always defined and degrades cleanly. \\
\hline
RAG generation blocking the FastAPI event loop & Each RAG suggestion call in the request handler runs a sentence-transformer embedding of the student's recent submissions plus an OpenAI completion --- both synchronous and slow enough that, if executed inline in the FastAPI event loop, would block every concurrent request until the call returned. & The RAG router \texttt{await}s \texttt{asyncio.to\_thread(\ldots)} around the generator call, offloading the synchronous work to the default thread pool so the event loop stays responsive. The first-time ChromaDB collection build is a separate concern handled by the \texttt{lifespan} startup prewarm, not by the request handler.  \\
\hline
\end{longtable}
